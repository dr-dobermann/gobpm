## §1 Симптомы

### §1.1 Exclusive-разветвление по первому-истинному возвращает случайного победителя при перекрытии условий

Когда два исходящих потока exclusive-шлюза оба имеют истинное условие, правило
«первый-истинный» (ADR-005 §2.8) должно выбрать объявленный первым. Вместо этого
оно выбирает произвольный, варьируясь от прогона к прогону. Регрессионный тест
`TestExclusiveGatewayExec/first_true_wins_on_overlap` (`exclusive_test.go`)
вследствие этого нестабилен (flaky):

```
--- FAIL: TestExclusiveGatewayExec/first_true_wins_on_overlap
    exclusive_test.go:188:
        []*flow.SequenceFlow{0xc0000c7e00} does not contain &flow.SequenceFlow{...id:"…"}
```

6/30 прогонов падают. В коде: `ExclusiveGateway.Exec` (`exclusive.go`) обходит
`eg.Outgoing()` и возвращает первый поток, чьё условие истинно — но
`Outgoing()` не имеет определённого порядка (см. §2), так что «первый» случаен.
BPMN §13.4.2 требует, чтобы условия «вычислялись в [объявленном] порядке»;
приземление не конформно.

## §2 Анализ первопричины

### §2.1 `Outgoing()`/`Incoming()` обходят карту → неопределённый порядок

`BaseNode` хранит потоки в карте и возвращает значения карты:

```go
// pkg/model/flow/node.go:98
flows map[data.Direction]map[string]*SequenceFlow
// :137
func (fn *BaseNode) Incoming() []*SequenceFlow { return maps.Values(fn.flows[data.Input]) }
// :142
func (fn *BaseNode) Outgoing() []*SequenceFlow { return maps.Values(fn.flows[data.Output]) }
```

`maps.Values` по Go-карте выдаёт записи в **рандомизированном** порядке
итерации. Так что `Outgoing()` возвращает разный порядок при каждом
вызове/прогоне. Затронут каждый потребитель, которому важен порядок — прежде
всего **Exclusive-разветвление по первому-истинному** (§2.8) выше; подмножество
**Inclusive** (§2.9) и OR-join (SRD-022) тоже обходят потоки и выигрывают от
стабильного порядка. `AddFlow` (`node.go:146`) фиксирует вставку, но карта
отбрасывает этот порядок.

### §2.2 Тест утверждает конкретного победителя → flaky, а не жёсткий сигнал

`first_true_wins_on_overlap` связывает поток A, затем поток B (оба истинны) и
ожидает A. Он проходит только когда карта случайно выдаёт A первым — так что
дефект всплыл как перемежающаяся нестабильность CI, а не как детерминированное
падение, и проскользнул мимо приземления SRD-021.

## §3 Решение

### §3.1 Рассмотренные альтернативы

| Альтернатива | За | Против | Решение |
|---|---|---|---|
| A. Сортировать `Outgoing()` по id потока | детерминированно | **не порядок объявления** — BPMN §13.4.2 хочет объявленный порядок; id — не объявление | ❌ отклонено |
| B. Один срез на направление в порядке объявления (`map[Direction][]*SequenceFlow`) | одна структура, естественно упорядоченная, безопасная к перезаписи; `Outgoing` — единственный clone без по-flow-индирекции | дедупликация O(n) при добавлении — но у узла лишь горстка потоков, так что это неважно | ✅ выбрано |
| C. Оставить id-карту + параллельный срез порядка `[]id` | дедупликация O(1) | **две структуры держать в синхроне**; `Outgoing` тогда платит поиск по карте на каждый поток ради восстановления порядка; единственная оставшаяся работа карты — проверка дедупликации | ❌ отклонено (избыточно — дедупликация O(1) ничего не даёт на таком масштабе) |

### §3.2 Изменения по файлам

#### §3.2.1 `pkg/model/flow/node.go` — потоки в порядке объявления

Хранить потоки каждого направления в одном **срезе в порядке объявления**.
`Outgoing()`/`Incoming()` возвращают защитную копию (`slices.Clone`, оставляя
внутренний срез не-экспонированным, как делал `maps.Values`). `AddFlow`
дописывает новый поток или перезаписывает на месте, когда его id уже присутствует
(сохраняя его позицию). Убрать внутреннюю id-карту и теперь-неиспользуемый импорт
`golang.org/x/exp/maps` (добавить `slices`).

```go
// before:
type BaseNode struct {
	flows map[data.Direction]map[string]*SequenceFlow
	BaseElement
}
func (fn *BaseNode) Outgoing() []*SequenceFlow { return maps.Values(fn.flows[data.Output]) }

// after:
type BaseNode struct {
	flows map[data.Direction][]*SequenceFlow
	BaseElement
}
func (fn *BaseNode) Outgoing() []*SequenceFlow { return fn.ordered(data.Output) }
// ordered returns a copy of the direction's flows, already in declaration order.
func (fn *BaseNode) ordered(dir data.Direction) []*SequenceFlow {
	return slices.Clone(fn.flows[dir])
}
```

`AddFlow` сканирует срез направления: при совпадающем id перезаписывает на месте;
иначе дописывает (`append` также покрывает случай nil-среза, так что прежний шаг
«создать внутреннюю карту, если отсутствует» исчез).

## §4 Верификация

Текущее покрытие в каталоге тестов:
- unit: добавление/получение потоков `node` протестировано, но **нет утверждения
  о порядке** (это и есть пробел).
- тест шлюза first-true существует, но flaky (это симптом).

### §4.1 Регрессионные тесты (обязательны)

#### §4.1.1 `TestBaseNodeFlowOrder`

**Новое:** `pkg/model/flow/node_test.go` (или существующий файл тестов node).

| Тест | Установка | Утверждение |
|---|---|---|
| `TestBaseNodeFlowOrder` | добавить потоки f1, f2, f3 (in/out) | `Outgoing()`/`Incoming()` возвращают их в **порядке добавления**, стабильно при повторных вызовах; повторное добавление id сохраняет его позицию и обновляет значение |

#### §4.1.2 Устранить нестабильность теста шлюза

`TestExclusiveGatewayExec/first_true_wins_on_overlap` становится
детерминированным (порядок объявления ⇒ первый связанный поток всегда
побеждает): `go test -run … -count=50` → **0 падений**.

## §5 Предотвращение

- Doc-комментарий к `Outgoing()`/`Incoming()` с **гарантией порядка объявления**
  и заметкой, что правила шлюза first-true/subset зависят от него.
- `TestBaseNodeFlowOrder` — канарейка: если он падает, упорядочивание
  регрессировало и разветвления шлюзов снова стали недетерминированными.

## §6 Регрессии / побочные эффекты

### §6.1 Что может полагаться на старое поведение

Ничто сегодня не может полагаться на *конкретный* порядок (он был случайным).
Потребители либо обходят все потоки (порядок-агностично), либо хотят
first-true (шлюзы — исправлены этим). Audit:
`grep -rn '\.Outgoing()\|\.Incoming()' pkg internal` — подтвердить, что каждое
попадание порядок-агностично или это шлюз. Поведенческой регрессии не ожидается;
изменение только *добавляет* гарантию.

### §6.2 Путь отката

Откат одним коммитом `node.go` (+ тест).

### §8.2 Эмпирические находки — где реальность разошлась с черновиком §3

**Один упорядоченный срез, не карта+порядок.** Черновик §3.1 сначала выбрал
карту плюс параллельный срез порядка (вариант B); при ревью это оказалось
избыточным (две структуры для синхронизации плюс поиск по карте на каждый поток
в `Outgoing`). Перед приземлением переработано в единый
`map[Direction][]*SequenceFlow` — `ordered` это один `slices.Clone`.
§3.1/§3.2 исправлены, чтобы отразить это.

**Результат.** `first_true_wins_on_overlap` прошёл путь от 6/30 flaky до 0/50;
`make ci` зелёный (diff-coverage 100% на затронутых строках).
