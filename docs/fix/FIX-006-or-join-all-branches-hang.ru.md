## §1 Симптомы

Inclusive (OR) ромб, где условие **каждой** ветви разветвления истинно — так что все
ветви форкаются и приходят в join — **зависает**: инстанс никогда не завершается
(ограниченный по времени `WaitCompletion` истекает по таймауту).

```
start → OR-split ─┬[x<100]→ a ─┐
                  ├[x>10] → b ─┼ OR-join → end      (x = 50 ⇒ все три истинны)
                  └[x>=0] → c ─┘
```

Случаи OR-join, где взято *часть-ветвей* (срабатывание через достижимость), всегда
работали; зависал только случай *все-взяты* — для которого не было теста (все тесты
SRD-022 оставляют хотя бы одну ветвь невзятой).

## §2 Анализ первопричины

Два различных дефекта, оба — «запаркованный трек никогда не будится»:

### §2.1 Срабатывание по завершении прихода никогда не сигналит запаркованным слитым трекам

OR-join завершается двумя путями. Срабатывание по *достижимости* идёт через
`Instance.fireOrJoin`, который сигналит `parkCh` каждого запаркованного трека.
Срабатывание по *завершению прихода* (последний приход помечает каждый входящий поток →
`InclusiveGateway.Arrive` возвращает `complete` → `track.synchronize` испускает `evMerged`)
идёт через `Instance.applyMerged`, который переводил более ранние (запаркованные,
`AwaitSync`) приходы в `Merged`, **но никогда не сигналил их `parkCh`**. Так что при всех
взятых ветвях последний приход завершает join, а более ранние два зависают на `parkCh`.

Первая попытка сигналила `parkCh` только когда слитый трек уже был в `AwaitSync` — но это
**гонка с собственным переходом трека** в `AwaitSync`: трек возвращается из `Arrive`
(false), а цикл может обработать завершающий `evMerged` до того, как трек достигнет
`updateState(AwaitSync)`, так что гейт видит «не запаркован» и пропускает его.

### §2.2 Поздний приход в уже-сработавший join паркуется, а не поглощается

Срабатывание по достижимости может завершить join раньше, чем ветвь, которую оно сочло
недостижимой, реально придёт (временно́е окно). Токен этой ветви затем доходит до join,
`Arrive` возвращает «не complete» (join уже `fired`), и `synchronize` **паркует** его на
`AwaitSync` — где его ничто никогда не разбудит (его нет в сработавшем `order`, так что ни
одно срабатывание ему не сигналит). У OR-join не было обработки **хвостового токена** —
ровно того, что есть у Complex-шлюза (`Record` → `firedAlready`,
[SRD-023](../srd/SRD-023-complex-gateway.ru.md)).

## §3 Решение

### §3.1 Рассмотренные альтернативы

| Альтернатива | Решение |
|---|---|
| Сигналить слитому `parkCh` только когда `AwaitSync` (первая попытка) | ❌ гонка с переходом трека в `AwaitSync` (§2.1) |
| Добавить `firedAlready` в `SynchronizingJoin.Arrive` | ❌ переворошит контракт, общий с Parallel, и каждое место вызова `Arrive` — ради OR-only заботы |
| `Fired()`, затем отдельная проверка хвоста вне цикла | ❌ гонка: запаркованный in-order трек и хвостовой оба видят «fired» |
| **`applyMerged` будит слитые безусловно + `ReachabilityJoin.IsTrailing` поглощает поздние приходы** | ✅ выбрано — без гонок, ограничено OR |

### §3.2 Изменения по файлам

#### §3.2.1 `internal/instance/instance.go` — `applyMerged` будит слитые безусловно

`applyMerged` теперь сигналит `parkCh` каждого слитого трека **безусловно** (без гейта по
`AwaitSync`): запаркованный трек возобновляется и возвращается; ещё не достигший park-`select`
прочитает буферизованный(1) сигнал, когда заблокируется; Parallel-трек `AwaitingMerge`
(горутина уже вернулась) просто никогда его не прочитает. Это без гонок и покрывает оба пути
срабатывания. `fireOrJoin` отбрасывает свой теперь-избыточный цикл по слитым (оставляет только
сигнал выжившему).

#### §3.2.2 `pkg/exec/exec.go` + `pkg/model/gateways/inclusive.go` — `IsTrailing`

`ReachabilityJoin` получает `IsTrailing(arrivingTrackID string) bool`;
`InclusiveGateway.IsTrailing` возвращает `fired && !slices.Contains(order, trackID)`
под мьютексом шлюза (так что ответ согласован с собственным `Arrive` трека).

#### §3.2.3 `internal/instance/instance.go` — `recheckParked`

Ветвь `evParked` цикла теперь вызывает `recheckParked(track)`: если для этого трека join
`IsTrailing` (уже сработал, не записав его), трек поглощается — переводится в `Merged` и
будится, чтобы его горутина вернулась — вместо вечного ожидания; иначе перепроверяет join как
раньше.

## §4 Верификация

- **`TestORJoinAllBranchesArrive`** (`pkg/thresher/or_join_test.go`): OR-ромб со всеми тремя
  взятыми ветвями завершается один раз и прогоняет все три ветви. Под `-race` ×25 — 0
  зависаний (было ~20–30 %).
- Полный набор тестов join (`internal/instance`, `pkg/thresher`, `pkg/model/gateways`)
  зелёный под `-race`. Parallel и Complex не затронуты (§6).
- **Наблюдаемость:** debug-уровневое логирование track-событий (тот же PR) делает этот класс
  зависаний само-очевидным — события `parked` без последующих `merged`/`ended`.

## §5 Предотвращение

- Doc-комментарии к `applyMerged` / `IsTrailing` / `recheckParked` объясняют гонку и почему
  пробуждение безусловно.
- Регрессионный тест на случай «все-взяты» — канарейка; логирование событий — постоянная
  наблюдаемость для «join, который никогда не срабатывает».

## §6 Регрессии / побочные эффекты

- **Parallel**-шлюз: не затронут — это `SynchronizingJoin`, а не `ReachabilityJoin`, так что
  `IsTrailing`/`recheckParked` к нему никогда не применяются; безусловное пробуждение в
  `applyMerged` безвредно для его треков `AwaitingMerge` (уже-вернувшихся).
- **Complex**-шлюз: не затронут — он поглощает хвостовые токены сам в
  `synchronizeActivation` (`Record` → `firedAlready`), так что никогда не паркует их.
- Откат: реверт одним коммитом.
