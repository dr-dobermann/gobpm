Этот SRD специфицирует приземление **Group-A**, приводящее `internal/instance/` к двухслойному рантайму из [ADR-001 v.3](../design/ADR-001-execution-model.md): `track` — операционный поток исполнения; `token` становится **логической проекцией** текущего шага track'а, а не хранимым типом; состояние instance мутируется только внутри единственной горутины event-loop'а. Завершение этого приземления закрывает приёмочный гейт §7 ADR-001 v.3.

## 1. Предпосылки и мотивация

### 1.1 Текущее состояние (сверено с кодом)

Текущий `internal/instance/` реализует трёхслойную модель, которую заменяет ADR-001 v.3:

- **`token` — хранимый перекрёстно-связанный объект** — `type token struct { inst *Instance; trk *track; …; prevs []*token; nexts []*token; state TokenState }` (`token.go:36`). Имеет обратные ссылки и на Instance, и на свой track.
- **Instance владеет реестром токенов** — `tokens []*token` (`instance.go:101`) под `sync.RWMutex` (`instance.go:103`); `newToken` самрегистрируется через `inst.addToken` (`token.go:67`, `instance.go:440`).
- **Нижний слой ведёт жизненный цикл instance** — `token.updateState` вызывает наверх `t.inst.tokenConsumed(t)`, когда токен потребляется (`token.go:81`, `instance.go:452`); `tokenConsumed` сканирует реестр токенов на любой `TokenAlive` и останавливает instance, когда таких не осталось.
- **Форк держит токены на одном track'е, затем переназначает** — `token.split(n)` создаёт N токенов, привязанных к тому же track'у (`newToken(t.inst, t.trk)`, `token.go:89–95`); `checkFlows` затем переназначает N−1 из них свежесозданным track'ам (`track.go:454–512`, split в `track.go:470`).
- **У track'а нет единого токена** — текущий токен живёт в `stepInfo.tk` (`track.go:140–142`); track — `type track struct { …; steps []*stepInfo; m sync.RWMutex; state trackState }` (`track.go:148`).
- **Мутация состояния lock-based и реактивна** — track'и вызывают методы Instance напрямую (`addTrack`, вызывающие из `track.go`) под `RWMutex`.
- **Происхождение дублировано** — на track'е (`track.prev`) и на токене (`token.prevs/nexts`).
- **Enum состояния токена** — `TokenInvalid / TokenAlive / TokenWaitForEvent / TokenConsumed` (`token.go:15–22`); без `Withdrawn`.

### 1.2 Почему меняем

[ADR-001 v.3 §3.1/§6](../design/ADR-001-execution-model.md) фиксирует решение: при 1:1 без миграции хранимый токен лишь дублирует track (вынужденные обратные ссылки, второй реестр, дублирующая цепочка происхождения), а lock-based реактивная модель — это класс гонок, от которого проект уходит (ср. FIX-001). Рефакторинг убирает тип token, делает токен проекцией и сериализует всю мутацию состояния instance через одну горутину event-loop'а.

## 2. Цели и область

### 2.1 Цели (в области)

G1. `token` больше не хранимая сущность — это значение, **проецируемое** из текущего шага track'а.
G2. `track` владеет ровно одной позицией+состоянием; выставляет её через `track.Token()`.
G3. Instance держит **только реестр track'ов** — без реестра токенов, без обратных ссылок `token→Instance`/`token→track`.
G4. Состояние instance мутируется **только** внутри одной горутины event-loop'а; track'и сообщают прогресс событиями в канал; без блокировок на состоянии instance.
G5. **Форк** конструирует новый track на каждый дополнительный активный flow (родитель продолжает по первому flow); происхождение на `track.prev`.
G6. **Завершение instance** решается по реестру track'ов (все track'и закончились), а не сканированием состояний токенов.
G7. **Состояние** токена — проекция состояния track/шага (+ причина завершения track'а).
G8. **Без изменения внешнего поведения** для уже реализованных элементов: None Start/End события, Service/User задачи, Exclusive-шлюз, sequence flow (вкл. условия/default) и timer-события исполняются как сейчас (существующие тесты + `examples/*` проходят без изменений).
G9. Instance может сообщать **историю пути token-flow** — упорядоченную последовательность узлов, пройденную каждым токеном — реконструируемую из track'ов (живых и завершённых) и их шагов.
G10. Шаги несут **временные метки**, так что история пути (G9) — это таймлайн (на узел: вход/выход, длительность). Время приходит из инъектируемого seam'а, совместимого с расширением `Clock` из ADR-002 (здесь не строится).

### 2.2 Не-цели (явно отложено)

N1. **Персистентность / регидрация** — состояние Scope/таймера/компенсации/ошибок/активности, контракт состояния на узел, чекпойнты, восстановление после рестарта → выделенный Persistence & State ADR (см. ADR-001 v.3 §4.7).
N2. **Вся семантика join/merge** → gateway SRD ([ADR-005](../design/ADR-005-gateways-and-joins.md)). Сюда входят синхронизирующие join'ы (политики Parallel / Inclusive / Complex, seam `JoinController` / `JoinerContext`, семантика OR-join) **и** сам **несинхронизирующий merge**. *Пересмотрено при приземлении:* seam join'а (изначально FR-12) отложен, чтобы строиться со своим первым реальным потребителем, а не с фейковым; несинхронизирующий merge (изначально FR-8, в области) был **также отложен**, когда оказался не безопасным к гонкам — узел, пересекаемый двумя track'ами (достижимо через неуправляемый split со схождением), даёт гонку на `NodeDataLoader.RegisterData`, мутирующем разделяемый узел (нарушение неизменяемости §4.7, исправляется контрактом состояния на узел в Persistence ADR). Group A поэтому даёт **форк** (FR-5), но не безопасный к гонкам merge. Дизайн политики/механизма §4.2 сохранён как опережающая концепция. См. §7 Implementation Summary и [ADR-001 v.3 §9](../design/ADR-001-execution-model.md).
N3. **Event-Based Gateway** и **производитель** `Withdrawn` → gateway SRD. Проекция состояния токена (G7) определяет `Withdrawn`, но в этом приземлении его ничто не производит.
N4. **Новые элементы BPMN** (Parallel/Inclusive/Complex шлюзы, под-процессы, корреляция сообщений и т.п.).
N5. **Промоут публичного API** — `Token` / `GetTokens` остаются в `internal/instance/`; перенос типов исполнения в `pkg/` — работа ADR-003 (module-layout).

## 4. Дизайн и план реализации

### 4.1 Целевые формы (иллюстративно)

```go
// Token — вычисляемый read-model, никогда не хранимый как живой перекрёстно-связанный объект.
type Token struct {
    Node  flow.Node    // текущая позиция во flow (указатель на узел, из snapshot)
    State TokenState   // спроецировано из состояния track/шага
    // происхождение выставляется через проекцию track.prev
}

// Track записывает каждый переход состояния шага записью в упорядоченный
// append-only список (записывать, не перезаписывать). Проекция токена, история
// пути и тайминг ВСЕ выводятся из этих списков — нет отдельно поддерживаемых
// структур проекции для синхронизации.
type stepUpdate struct {
    Node  flow.Node   // узел, исполнявшийся на этом шаге (указатель; snapshot держит все узлы)
    State trackState  // состояние track на этом переходе (источник проекции состояния токена)
    At    time.Time   // из инъектируемого time-seam (→ ADR-002 Clock позже)
}
// каждый track держит: updates []stepUpdate  +  track.prev (происхождение форка). Track владеет
// этим списком (Вариант B); живые чтения текущей позиции обслуживаются lock-free атомарным снимком.

// события track→loop (контракт loop'а) несут track + УКАЗАТЕЛЬ на узел, не строковый ID.
// Построено в Group A:
//   Fork(track, active []*flow.SequenceFlow)   // evFork — loop строит по одному новому track'у на доп. flow
//   Ended(track)                               // evEnded — run() track'а вернулся
// Отложено:
//   JoinArrival(...)  → gateway SRD (синхронизирующие join'ы)
//   Wait(...)         → persistence ADR (освобождение на длинном ожидании)

func (t *track) Token() Token          // выводимо: последнее обновление (позиция + спроецированное состояние токена)
func (i *Instance) GetTokens() []Token // выводимо: последнее обновление каждого АКТИВНОГО track'а
func (i *Instance) TokenHistory() ...  // выводимо: списки обновлений всех track'ов (живых + завершённых),
                                       // сшитые по track.prev — вызывающий читает путь, тайминг
                                       // и терминальное состояние прямо из записей

// Seam политики join (§4.2) — ОТЛОЖЕНО в gateway SRD (пересмотрено при приземлении,
// см. §7). Дизайн ниже сохранён как опережающая концепция: loop владеет
// механизмом (учёт + действие); join-узел поставляет политику как ЧИСТЫЙ метод;
// Instance реализует JoinerContext, передавая политике ФАКТЫ (не решения).
// Строится со своим первым синхронизирующим потребителем, не сейчас.
type JoinerContext interface {        // реализуется Instance; растёт по мере нужд политик
    JoinNode() flow.Node              // оцениваемый join
    Arrived() []*flow.SequenceFlow    // входящие flow, держащие токен сейчас
    // (в будущем: позиции/история живых track'ов, для правила достижимости Inclusive)
}
type JoinVerdict struct {
    Fire     bool                  // false => продолжать ждать
    Consumed []*flow.SequenceFlow  // входящие flow, потреблённые при срабатывании
}
type JoinController interface {        // опционально; синхронизирующие узлы реализуют
    EvaluateJoin(ctx JoinerContext) JoinVerdict
}
// loop: узел реализует JoinController -> консультируемся; иначе -> срабатываем сразу (pass-through)

type Instance struct {
    ctx    context.Context
    cancel context.CancelFunc
    trackEvents chan trackEvent   // tracks -> loop()
    external    chan ExternalSignal
    tracks map[string]*track      // мутируется ТОЛЬКО в loop()
    state  InstanceState
    // нет tokens[]; нет RWMutex на состоянии instance
}
```

### 4.2 Механика форка и join (event-driven)

И форк, и join — реакции `Instance.loop()` на события, эмитируемые track'ами — track никогда не мутирует реестр instance сам.

**Форк** — узел с N>1 активными исходящими flow (Parallel/Inclusive split или активность с несколькими исходящими flow = неуправляемый split):

1. Исполняющий track определяет свои активные исходящие flow F1…FN (выбор, *какие* flow активны — дело шлюза/узла; механика ниже действует на уже выбранном множестве) и эмитит одно **событие форка**, несущее их плюс свой ID (для происхождения).
2. Track продвигает свою позицию на цель F1 и продолжает — он **не** заканчивается на форке.
3. `loop()` получает событие и для каждого оставшегося flow F2…FN конструирует новый track на цели этого flow с `track.prev` = форкающий track, регистрирует и порождает его горутину.
4. Итог: N track'ов работают независимо (исходный на F1 + N−1 новых), каждый со своей позицией и горутиной.

**Join** — узел с N>1 входящими flow:

1. Track, прибывающий на join, эмитит **событие прибытия на join**. Для несинхронизирующего merge затем просто продолжает; для синхронизирующего join заканчивает свою горутину в ожидании решения loop'а.
2. `loop()` координирует по типу merge:
   - **Несинхронизирующий** (Exclusive merge, неуправляемый merge) — единственный вид в области Group-A: без ожидания, без потребления; каждый прибывающий track продолжает по исходящему flow независимо (несколько track'ов могут законно пересекать один узел).
   - **Синхронизирующий** (Parallel/Inclusive) — вне области Group-A (gateway SRD): loop держит прибытия, пока ожидаемое множество не полно, затем один track продолжает, остальные заканчиваются (`TrackMerged`).
3. На join новый track не создаётся — продолжение всегда едет на прибывшем track'е.

**Владение координацией join.** Синхронизация flow на join — ответственность **Instance**, не узла. Per-join состояние рантайма (какие входящие flow прибыли) — нужное только синхронизирующим join'ам — живёт в `loop()`, ключом по ID узла, мутируется только там (один писатель, lock-free). Определение узла остаётся **неизменяемым и разделяемым** между instance и track'ами, поэтому никогда не держит состояние прибытия — положить его на узел означало бы вернуть межтрековую гонку shared-mutable-state, которую этот дизайн убирает. Событие прибытия на join — точка подключения, где логика синхронизирующего join (gateway SRD) встроится.

**Политика vs механизм — кто решает join.** Instance владеет *механизмом*: держит межтрековый учёт прибытий и выполняет действие (продолжить один track, merge/завершить остальные). *Политика* — «полно ли ожидаемое множество и какие прибытия потреблены?» — принадлежит join-**узлу** (его тип/конфиг шлюза), потому что только узел знает семантику join и только граф процесса определяет ожидаемое входящее множество. Прибывающий track решить не может (он знает только себя; он лишь сообщает). Так что на каждое прибытие `loop()` **консультирует узел через чистый, read-only `JoinController.EvaluateJoin`**, передавая ему `JoinerContext`, реализуемый Instance — **только факты**: прибывшие входящие flow сейчас, плюс позиции/история живых track'ов для правила Inclusive позже. Политика вычисляет вердикт — *ждать* или *сработать* с потреблёнными flow — и loop действует по нему. Факты, не решения: неоднозначная достижимость Inclusive вычисляется внутри политики из этих фактов, никогда loop'ом. Узел ничего не хранит; это чистая политика, оцениваемая против состояния, поставляемого Instance (неизменяемость узла сохранена). Следовательно ожидаемый счётчик **не** захардкожен в Instance: для Parallel join это статические входящие flow узла; для Inclusive join политика узла выводит его из живого исполнения.

**Отложено при приземлении (§7):** seam join — интерфейс `JoinController`, механизм consult-or-default и per-join учёт — **не** строится в Group A. Синхронизирующего потребителя пока нет (Parallel/Inclusive — gateway-SRD), поэтому он строится там, со своим первым реальным потребителем, а не с фейковым тестом сейчас. Несинхронизирующий merge (FR-8) **также отложен** — он не безопасен к гонкам в этом приземлении (мутация разделяемого узла `RegisterData`; см. §7). Общий дизайн (`arrivals + context → wait | fire(consumed)`) выше стоит как опережающая концепция; построение его позже не втягивает отложенную семантику OR-join — та живёт внутри будущей реализации политики.

### 4.3 Изменения по областям

- **`token.go`** — убрать хранимую структуру `token` (обратные ссылки, `state`, `prevs/nexts`), вызов `addToken` в `newToken`, вызов `tokenConsumed` в `updateState` и `split`. Оставить `TokenState` как проецируемый enum (добавить значение `Withdrawn`, без производителя пока). Добавить значение-проекцию `Token`.
- **`track.go`** — track владеет своей позицией+состоянием напрямую; добавить `Token()`, проецирующий текущий шаг; добавить **причину завершения** track'а (различать withdrawn vs canceled, FR-7); переработать `checkFlows` на **эмиссию события форка** вместо `split`-затем-переназначение — loop конструирует новые track'и (§4.2, FR-5); происхождение остаётся на `track.prev`; записывать каждый переход состояния шага как запись с меткой времени в append-only список (записывать, не перезаписывать — FR-10/11), штампуя из инъектируемого time-seam (→ ADR-002 `Clock`) — `Token()` / `GetTokens()` / `TokenHistory()` и весь тайминг выводятся из этих списков; убрать использование `RWMutex` состояния instance в пользу эмиссии событий.
- **`instance.go`** — убрать `tokens[]`, `addToken`, `tokenConsumed`; добавить горутину `loop()` + топологию каналов (FR-4) как единственный мутатор состояния и **координатор форка/join** (§4.2); перевести `addTrack`/порождение track'а/форк/прибытие-на-join/завершение в события, применяемые в `loop()`; решать завершение по реестру track'ов (FR-6); добавить `GetTokens()` и `TokenHistory()` (FR-10). **Сохранять завершённые track'и** (их списки обновлений + происхождение + причину завершения) на время жизни instance, чтобы история реконструировалась; реестр различает активные vs завершённые, и завершение (FR-6) опирается на активные track'и, не на удаление записей. (Неограниченное удержание под циклами / мульти-instance — позднейший концерн; эти элементы вне области здесь; долговечная/ограниченная история принадлежит Persistence & State ADR.)
- **Моки** — регенерировать через `make gen_mock_files`, если меняется сигнатура замоканного интерфейса.

### 4.4 Вехи (каждая независимо собираема и CI-зелёная)

1. **M1 — инфра: time-seam + тест-хелперы.** Инъектируемый внутренний time-seam (→ ADR-002 `Clock` позже); фейковые часы + хелпер assert'а утечки горутин. (Источник FR-11.)
2. **M2 — event loop Instance + сериализованная мутация.** `Instance.loop()` + топология каналов становится единственным мутатором состояния instance и координатором форка; существующий путь форка перенаправлен через событие (здесь ещё на `split`); снять `RWMutex` с состояния instance. (FR-4)
3. **M3 — список состояний шага + проекции.** Append-only список обновлений состояния шага с метками времени; выводы `track.Token()` / `Instance.GetTokens()` / `TokenHistory()` + проекция состояния токена, сосуществующие с текущим типом token. (FR-2, 7, 10, 11)
4. **M4 — переработка форка.** Обработчик события форка в `loop()` конструирует по одному новому track'у на доп. активный flow (родитель продолжает по F1, происхождение `track.prev`); убрать `token.split`; новые track'и сами создают токен при исполнении (тип token ещё есть). (FR-5)
5. **M5 — удаление типа token.** Убрать хранимый тип `token` + `Instance.tokens` / `addToken` / `tokenConsumed` / `token.inst` / `trk`; состояние токена чисто через проекцию M3; завершение решается по активным track'ам. (FR-1, 3, 6)
6. **M6 — приёмочная верификация.** Полный набор §7 ADR-001 v.3 + race-стресс (`-count=N`) + examples + покрытие; `make ci` зелёный. (FR-9, NFR-1..3) *(Несинхронизирующий merge FR-8 отложен при flip'е в Accepted — не безопасен к гонкам; см. §7.)*

Обоснование последовательности: сначала поднять event loop (M1/M2), чтобы изменения владения (M4/M5) приземлялись на фундаменте сериализованной мутации, а не гонялись со старыми lock-путями; проекции (M3) предшествуют удалению типа token (M5), чтобы проекция была источником состояния токена до удаления типа.

## 5. Верификация (Definition of Done)

Отображается на [ADR-001 v.3 §7](../design/ADR-001-execution-model.md):

| Тест | Утверждает |
|---|---|
| Свобода от гонок | `-race` с гейтом CI; состояние instance мутируется только в `loop()`; `-race -count=N` стресс на форк/instance чистый. |
| Без утечки горутин | `runtime.NumGoroutine()` возвращается к базовой линии после завершения. |
| Форк 1:1 | split → независимые track'и, каждый со своей позицией; родитель продолжил по первому flow (`TestM4ForkCompletes`). |
| Нет реестра токенов | Instance выставляет токены только через `GetTokens()`; поля `tokens` нет; `track.Token()` отражает текущий шаг. |
| Завершение instance | `Completed` тогда и только тогда, когда все track'и закончились (без сканирования живых токенов). |
| ~~Несинхронизирующий merge~~ | **ОТЛОЖЕНО** — FR-8 → gateway SRD ([ADR-005](../design/ADR-005-gateways-and-joins.md)); не безопасен к гонкам здесь (§7). Теста merge в Group A нет. |
| Проекция состояния токена | каждое состояние track/шага отображается в ожидаемое `Token.State` (`TestTokenStateProjection`). |
| Каскад завершения | отмена ctx → все горутины track'ов выходят; instance достигает терминального состояния (`TestTerminationCascade`). *Триггер* Terminate End Event — [ADR-006](../design/ADR-006-events-and-subscriptions.md). |
| История пути токена | После завершения форкнутого прогона `TokenHistory()` возвращает общий до-форковый путь и по одному пути на ветку, каждый с верным происхождением и терминальным состоянием. |
| Тайминг шагов | Метки истории монотонно неубывают; под инъектированными фейковыми часами значения вход/выход на узел детерминированы. |
| Без регрессий | существующие тесты `internal/instance` + движка и `examples/*` проходят без изменений. |

**DoD:** все FR/NFR выполнены; таблица выше зелёная; `make ci` зелёный; покрытие ≥ базовой линии. Завершение этого DoD закрывает приёмочный гейт §7 ADR-001 v.3 — flip ADR-001 Draft → Accepted — **отдельный** follow-up (не часть этого SRD).

## 6. Риски и регрессии

- **Переписывание event-loop инвазивно.** Риск дедлока или утёкших горутин. Митигация: `defer`-очистка с эмиссией терминального события track'а; тест утечки NFR-2; поэтапные вехи (M1/M2 до M3/M4).
- **Смена семантики завершения** (реестр track'ов vs сканирование живых токенов). Риск: instance'ы, которые не завершаются или завершаются рано. Митигация: тест FR-6 + существующий `TestMonitoring` (`instance_test.go`) должны проходить.
- **Обработка событий во время ожиданий.** Текущие ждущие track'и используют `ProcessEvent`; модель повторного порождения в event-loop (ADR-001 §4.7) должна сохранить доставку сигналов EventHub для ожиданий timer/message. Митигация: тесты timer-событий + `examples/simple-timer`, `examples/timer-event` проходят.
- **Источник времени.** Прямое использование `time.Now()` нетестируемо и обходит будущее расширение `Clock`. Митигация: один инъектируемый внутренний time-seam сейчас (детерминированные тесты), заменяемый на расширение `Clock` из ADR-002 в WS-B — не разбрасывать вызовы `time.Now()`.
- **Дрейф моков/интерфейсов.** Митигация: регенерировать моки; `tidy-check-all`.
- **Расползание области в отложенные участки.** Синхронизирующие join'ы / персистентность / новые шлюзы — явные N-цели; держать их вне.

| M4+M5 | `f86d592` | переработка форка (без `split`); удаление типа `token`, `Instance.tokens`, `addToken`, `tokenConsumed`, `stepInfo.tk` (FR-1, 3, 5) |

**Итог:** двухслойный рантайм (Instance + track); «token» — чисто выводимая
проекция. Файлы: `internal/instance/{instance,track,token}.go` + тесты
`{clock,leakcheck,m2_loop,m3_projection,m4_fork}_test.go`.

**Отклонения от спеки (разрешены при приземлении):**

- **M4 и M5 объединены.** Рабочий форк требует завершения по track'ам, что
  требует убрать остановку по токенам (`tokenConsumed`), что каскадирует к
  удалению всего типа token — поэтому две вехи это один change-set.
- **FR-12 (seam join) отложен** в gateway SRD (N2): seam `JoinController`
  только с фейковым потребителем отброшен в пользу построения со своим первым
  синхронизирующим шлюзом. §4.1/§4.2 сохраняют дизайн как опережающую концепцию.
- **FR-8 (несинхронизирующий merge) тоже отложен** (пересмотрено при flip'е в
  Accepted, вскрыто `/check-srd`): он никогда не покрывался тестом, и написание
  такого теста показало, что он **не безопасен к гонкам** — узел, пересекаемый
  двумя track'ами (достижимо через неуправляемый split со схождением), даёт
  гонку на `EndEvent.RegisterData`, мутирующем **разделяемый узел** (нарушение
  неизменяемости §4.7). Исправление — контракт состояния на узел, принадлежащий
  Persistence ADR; сама семантика merge переходит в gateway SRD
  ([ADR-005](../design/ADR-005-gateways-and-joins.md)). Group A даёт форк (FR-5),
  не безопасный к гонкам merge. Это держит SRD-001 согласованным с
  [ADR-001 v.3 §7/§9](../design/ADR-001-execution-model.md), чей приёмочный гейт
  также не заявляет несинхронизирующий merge.
- **`stepUpdate` записывает `trackState`** (источник проекции состояния токена),
  не `stepState`, как первоначально показывал набросок §4.1.
- **Построены события: только `evFork` + `evEnded`**; `JoinArrival`
  (синхронизирующие join'ы) и `Wait` (освобождение на длинном ожидании) отложены
  в gateway SRD / persistence ADR.
- **Добавлены `inst.lastErr` / `LastErr()`** для записи фатальной ошибки
  построения форка (не было в исходном наброске).
- **До-прогонный `Token()` возвращает нулевую проекцию** — начальный seed
  `TrackReady` убран; история записывается, когда track запускается (так что
  использует работающие часы и остаётся детерминированной под инъектированными).

**Верификация (M6):** `make ci` зелёный сквозь — tidy, lint (все 6 модулей),
build-all, race-тесты (все модули), govulncheck чистый; тесты форка/проекции
race-стресс на `-count=20`; без регрессий (существующий набор + `examples/*`).

**Статус:** **Принято** (2026-06-07). Реализация приземлена (M1–M6) и
пере-сверена с кодом через `/check-srd` перед flip'ом — PASS после того, как FR-8
(несинхронизирующий merge) отложен (выше), все прочие FR/NFR зелёные, `make ci`
зелёный, кросс-док-пины на ADR-001 v.3 согласованы. Русский twin добавлен по
правилу bilingual-on-Accepted.
