> EN-оригинал — канонический: [SRD-002-ci-diff-coverage-gate.md](SRD-002-ci-diff-coverage-gate.md). Этот файл — его перевод (twin).

Этот SRD предлагает автоматический **гейт diff-покрытия**: проверку, которая роняет CI, когда строки, которые изменение *добавляет или модифицирует*, недостаточно покрыты тестами — **без** блокировки на существующем backlog'е покрытия нетронутого кода. Он превращает проектную конвенцию «затронутые файлы заканчиваются на ≥80% (цель 100%)» из ручной дисциплины в механический, воспроизводимый гейт, стартуя с порога **70%** и повышая его со временем.

## 1. Предпосылки и мотивация

### 1.1 Текущее состояние (сверено с репозиторием)

- **`make ci`** (`Makefile:143`) = `tidy-check-all lint-all-modules build-all test-all vuln`. Зеркалит `.github/workflows/check.yml` (`check`, обязательный для защищённого `master`). Порога покрытия в нём **нет**.
- **Профили покрытия уже есть.** `test-all` (`Makefile:104-108`) запускает `go test -race -coverprofile=coverage.txt ./...` по модулю — каждый модуль уже создаёт `coverage.txt`.
- **Codecov уже подключён.** `check.yml` загружает покрытие в Codecov (`Upload coverage reports to Codecov`), у README есть бейдж Codecov.
- **Dev-инструменты под guard'ом.** Инструменты ставятся через `make tools` за guard'ом `require-tool` (правило fail-loud парности); Go-toolchain закреплён на `go1.25.11` в каждом `go.mod` и в workflow.

### 1.2 Проблема

Стандарт покрытия («каждый файл, который я создаю/обновляю, заканчивается на ≥80%, идеально 100%, вместе с изменением») живёт только в проектных конвенциях (CLAUDE.md / SDD-скиллы). Он держится на том, что автор *помнит* измерить — что и провалилось при финализации ADR-001: покрытие затронутых файлов проверили лишь после явного запроса. При этом **агрегат** ниже цели (например, `internal/instance` ~69%) — намеренно отложенный backlog.

Наивный гейт на агрегатный порог — неправильный инструмент: он блокировал бы на нетронутом backlog'е (или, выставленный низко под него, не enforce'ил бы ничего на новом коде). Нужен гейт **diff/patch-покрытия** — судить только **добавленные/изменённые строки**, оставив нетронутый legacy отдельному backlog'у.

## 2. Цели и область

### 2.1 Цели (в области)

- **G1.** Воспроизводимая проверка **diff-покрытия**: из строк, которые изменение добавляет/модифицирует (относительно базы PR / merge-base), покрытая доля должна быть ≥ настраиваемого порога, иначе проверка падает.
- **G2.** **Парность local↔CI.** Гейт запускается одним `make`-таргетом, так что «зелено локально ⇒ зелено в CI» сохраняется (ключевой этос проекта). Решение pass/fail не зависит от вычисления только на стороне сервера.
- **G3.** **Фазированный порог.** Один настраиваемый порог, **стартуя с 70%**, с документированным намерением повышать (→ 80 → к 100) по мере погашения backlog'а.
- **G4.** **Только область затронутых строк.** Нетронутый код (агрегатный backlog) явно вне гейта; он никогда не вызывает падение.
- **G5.** **Встроено в существующий гейт** (`make ci` + `check.yml`), инструмент ставится через `require-tool` + `make tools` (без тихого skip'а).
- **G6.** **Исключает не-продуктовый код** из измерения: `generated/` (моки), `*_test.go`, `examples/` и любые vendored/`doc.go`-only файлы.

### 2.2 Не-цели

- **N1.** Повышение **агрегатного** покрытия пакета — это отдельный отложенный backlog (`internal/instance` до ≥80%, широкий ревью). Гейт не должен с ним смешиваться.
- **N2.** Enforcement 100% сейчас — порог фазирован (G3).
- **N3.** Замена отчётности Codecov — Codecov остаётся для визуализации/трендов; этот гейт — про детерминированный pass/fail, воспроизводимый автором локально.
- **N4.** Гейты под стеки кроме Go — репозиторий Go-only; пересмотреть, если появится не-Go-модуль.
- **N5.** Мутационное тестирование / спор branch-vs-line — начинаем с line diff-покрытия.

## 3. Требования

| # | Требование | Приёмка |
|---|---|---|
| FR-1 | Таргет `make cover-check` вычисляет diff-покрытие из per-модульных профилей относительно base-ref'а и завершается ненулевым кодом ниже порога. | Запуск на ветке, добавляющей непокрытую функцию, падает; на полностью покрытом изменении — проходит. |
| FR-2 | Порог — одно настраиваемое значение (`Makefile`-переменная, напр. `COVER_MIN := 70`), документировано как «повышать к 80 → 100». | Смена одной переменной меняет гейт; дефолт 70. |
| FR-3 | Область diff = строки, изменённые относительно merge-base с базовой веткой (`origin/master` в CI; override `BASE` локально). | Изменение, трогающее только хорошо покрытые новые строки, проходит, даже если общий % файла низкий. |
| FR-4 | Измерение исключает `generated/`, `*_test.go`, `examples/` и не-продуктовые файлы. | Изменение в сгенерированных моках или тестах не влияет на гейт. |
| FR-5 | Встроено в `make ci` и `.github/workflows/check.yml`, чтобы `check` защищённого `master` это enforce'ил. | CI роняет PR, чьи изменённые строки ниже порога. |
| FR-6 | Инструмент diff-покрытия ставится через `make tools` за guard'ом `require-tool`; toolchain закреплён. | Отсутствие инструмента прерывает с install-подсказкой (без тихого pass'а); local и CI используют одну закреплённую версию. |
| FR-7 | CI делает checkout с достаточной историей для merge-base diff'а (`fetch-depth: 0` или эквивалент). | Base-diff резолвится в CI, не только локально. |
| NFR-1 | Парность local↔CI: один и тот же `make cover-check` решает pass/fail в обоих. | Pass/fail не зависит от вычисления только на сервере. |
| NFR-2 | Детерминированно и разумно быстро — переиспользует профили, которые `test-all` уже создаёт; без второго полного прогона тестов. | Wall-clock `make ci` растёт только на вычисление diff'а, не на повторный тест. |

## 4. Дизайн и альтернативы

### 4.1 Где запускается гейт — локальный таргет vs сервер (ключевое решение)

| Вариант | Механика | Вердикт |
|---|---|---|
| **A. Статус patch-покрытия Codecov** | Выставить в `codecov.yml` `coverage.status.patch.target: 70%`; Codecov (уже подключён) вычисляет patch (=diff) покрытие при загрузке и постит commit-статус; добавить этот статус в branch protection. | **Отклонено как *единственный* гейт.** Patch-покрытие — ровно то, что нужно, и усилий почти ноль — но pass/fail вычисляется **на стороне сервера при загрузке**, поэтому его **нельзя воспроизвести локально**. Это ломает несущее правило проекта «`make ci` зелено ⇒ CI зелено». Codecov остаётся для отчётности; гейтом его не делаем. |
| **B. Локальный `make cover-check` в `make ci`** — **выбрано** | `make`-таргет вычисляет diff-покрытие из существующих профилей `coverage.txt` относительно merge-base и падает ниже порога; встроен в `make ci` + `check.yml`. | **Выбрано.** Сохраняет парность local↔CI (G2/NFR-1); переиспользует профили, которые `test-all` уже создаёт (NFR-2); порог живёт в репозитории. Codecov остаётся дополнением для отчётности. |
| **C. Агрегатный порог** | Падать, если общее покрытие пакета < X%. | Отклонено. Блокирует на нетронутом backlog'е или не enforce'ит ничего (N1). Не diff-scoped. |

### 4.2 Инструмент для варианта B (закрепить при реализации)

Таргету нужно **покрытие изменённых строк** (patch) из Go-coverprofile + git diff. Кандидаты, решаемые на стадии milestone-плана по критериям *(должен работать локально; изменённые строки, не файл целиком; уважает excludes; поддерживаемый)*:

- **`gocovdiff`** — целевое покрытие изменённых строк относительно base-ref'а.
- **`octocov`** — отчёт покрытия + diff + пороги; CI-и-local; конфиг-файл.
- **`vladopajic/go-test-coverage`** — пороги file/package/total + режим diff-against-base; силён на excludes.
- **DIY-хелпер** — небольшой Go/awk-инструмент: парсит `coverage.txt` + `git diff --unified=0 <base>...HEAD`, пересекает изменённые строки с покрытыми, считает долю. Нулевая внешняя зависимость; больше кода в собственность.

**Решено (при реализации): DIY pure-Go хелпер.** Небольшая stdlib-only программа (`internal/covercheck` для тестируемой логики + `cmd/covercheck` для тонкого CLI) победила внешние инструменты: она не добавляет **никакой** зависимости (даже `go install`-бинарь, и тем более не Python/diff-cover), запускается одинаково локально и в CI (парность) и даёт точный контроль над формой multi-module + excludes. Правило minimal-deps (SAD G2) управляет hot-path движка, но держать *dev*-toolchain тоже лёгким — и избегать гейта только-на-сервере или на чужом языке — сделало ~250-строчный in-repo хелпер самым чистым вариантом.

### 4.3 Фазирование порога

`COVER_MIN` стартует с **70**. Повышать осознанно (своими крошечными change-set'ами): 70 → 80 (стоячий стандарт) → к 100 по мере погашения отложенного backlog'а. Каждое повышение — однострочное изменение, гейтится зелёным `make ci`.

### 4.4 Вехи

1. **M1 — таргет `make cover-check`** (локально), `COVER_MIN := 70`, excludes подключены, override base-ref'а; инструмент добавлен в `make tools` + `require-tool`. Доказать на образце под/над-покрытого изменения.
2. **M2 — встройка в CI**: добавить шаг в `check.yml` (с `fetch-depth: 0`) и в умбреллу `make ci`; подтвердить поведение на red/green PR.
3. **M3 — доки**: README «How we work» + заметка в CLAUDE.md, что порог diff-покрытия enforce'ится; зафиксировать план фазирования.

## 5. Верификация (Definition of Done)

| Проверка | Утверждает |
|---|---|
| Под-покрытое изменение падает | Ветка, добавляющая непокрытую функцию/строки, роняет `make cover-check` (и CI). |
| Покрытое изменение проходит | Ветка, чьи изменённые строки ≥ порога, проходит. |
| Backlog не блокирует | Трогание файла с низким агрегатом, но хорошо покрытыми новыми строками — проходит (diff-scoped, N1). |
| Excludes соблюдаются | Изменения в `generated/` / `*_test.go` / `examples/` не двигают гейт. |
| Парность | Один и тот же `make cover-check` решает pass/fail локально и в CI; `make ci` зелено ⇒ `check` зелено. |
| Guard инструмента | Отсутствие инструмента прерывает с install-подсказкой; закреплённая версия local == CI. |

**DoD:** FR-1..7 + NFR-1..2 выполнены; `make ci` зелено с активным гейтом на `COVER_MIN=70`; демонстрационный PR показывает, как гейт ловит под-покрытый diff; README + CLAUDE.md обновлены.

## 6. Риски и митигации

- **Shallow-clone в CI ломает merge-base.** → `fetch-depth: 0` (FR-7); явно проверить резолв base-diff'а в CI.
- **Flaky на первом PR / новых файлах.** Полностью новый файл — это 100% «изменённых строк» — он обязан соответствовать порогу как любой код; задокументировать, что новые файлы полностью в области.
- **Перекос от generated/моков.** → excludes (FR-4); проверить, что регенерация моков не роняет гейт.
- **Дрейф/заброс инструмента.** → закрепить версию в `make tools`; DIY-хелпер — запасной выход (без внешней зависимости).
- **Эрозия парности.** Никогда не оставлять решение гейта только в Codecov (отклонение §4.1); `make`-таргет — источник истины.

## 7. Итог реализации

Приземлено на ветке `refactor/adr-001-finalize` (рядом с финализацией ADR-001/SRD-001, по просьбе владельца держать одну ветку):

| Веха | Commit | Доставляет |
|---|---|---|
| Doc | `d401ec1` | этот SRD (draft) |
| M1 | `9b0e8bc` | `internal/covercheck` (ParseProfiles / ParseDiff / Evaluate / DefaultExcluded, 89.5% покрыто) + тонкий CLI `cmd/covercheck` + `make cover-check` (`COVER_MIN=70`, `COVER_BASE=origin/master`) |
| M2 | `0ee8933` | `cover-check` отвязан от `test-all` (переиспользует его `coverage.txt`, один прогон тестов); CLI громко падает без профиля; `check.yml` `fetch-depth: 0` + шаг «Diff-coverage gate» |
| M3 | _(этот commit)_ | заметки README + CLAUDE.md; этот итог; статус → Принято; RU twin |

**Выбранный подход:** DIY pure-Go хелпер (§4.2) — stdlib-only, ноль новых зависимостей, идентичный local/CI вердикт через один и тот же бинарь `cmd/covercheck`.

**Результат:** `make ci` запускает `test-all → cover-check` (один прогон тестов); CI запускает тот же `make cover-check COVER_BASE=origin/master` после шага тестов. Гейт **dogfood'ит сам себя** — на этой ветке он показывает `internal/covercheck.go` 85.7%, `instance.go` 72.2%, `track.go` 100% → **84.7% из 203 изменённых покрываемых строк, min 70% — PASS**. `cmd/` (CLI), `*_test.go`, `generated/` и `examples/` исключены.

**Отклонения от драфта:**

- **Инструмент закреплён на DIY pure-Go** (драфт оставлял выбор между gocovdiff / octocov / go-test-coverage / DIY). DIY выбран за нулевую зависимость + парность.
- **`cover-check` не зависит от `test-all`** (в драфте веха M1 предполагала зависимость): отвязка избегает второго `-race`-прогона в CI, где шаг тестов уже записал `coverage.txt`. CLI падает, если профиля нет, так что гейт не может пройти вхолостую.
- **Порог подтверждён на `COVER_MIN=70`** (`Makefile`-переменная, единый источник); CI-шаг передаёт `COVER_BASE=origin/master`.
