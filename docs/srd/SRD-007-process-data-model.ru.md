> EN-оригинал — канонический: [SRD-007-process-data-model.md](SRD-007-process-data-model.md). Этот файл — его перевод (twin).

Этот SRD приземляет [ADR-010](../design/ADR-010-process-data-model.md):
персистентные данные переезжают в per-instance **контейнерные scope'ы**,
которыми владеет выделенный компонент **data plane** с атомарностью целых
операций; каждое выполнение узла работает на **frame'е выполнения** с ключом
(track, node); узлы держат только иммутабельные определения данных плюс своё
lifetime-состояние по ADR-009. Он структурно упраздняет критическую гонку
данных §1.2 архитектурного аудита и снимает с Instance роль scope'а (аудит
§2.3, первый шаг).

## 1. Предпосылки и мотивация

### 1.1 Текущее состояние (сверено с кодом)

**Хранилище и блокировки сломаны четырьмя независимыми способами:**

- `Instance.addData` (`instance.go:505-532`) разрывает read-modify-write
  между захватами блокировки: поиск под `RLock` → освобождение →
  **мутация разделяемой внутренней карты без блокировки** → запись под
  `Lock`. Два трека, конкурентно добавляющие данные, теряют записи или валят
  рантайм (аудит §1.2).
- `Instance.getData` (`instance.go:537-571`) читает `inst.scopes` **вообще
  без блокировки**, включая цикл обхода родителей.
- `Instance.AddData` (`instance.go:716-738`) берёт `inst.m.Lock()` и затем
  вызывает `addData`, который берёт `inst.m.RLock()` — `RWMutex` в Go не
  реентерабелен, так что это **self-deadlock по построению**. Его
  единственный production-вызов — `UserTask.Exec` (`user_task.go:202`
  `re.AddData(ut, dd...)`); он живёт только потому, что каждый тест мокает
  scope (`user_task_test.go:93`).
- `Instance.ExtendScope` (`instance.go:812-842`) — это TOCTOU: проверка
  существования под `RLock`, сброс блокировки, вставка под `Lock`, затем
  `RegisterData` вне всякой блокировки.

**Модель данных выполнения противоречит сама себе:**

- Рабочие данные узла регистрируются под путём **с ключом-именем**
  (`/<process>/<nodeName>`) через `track.prepareNodeExecution` →
  `ExtendScope` → `RegisterData` (`track.go:506-516`, `instance.go:835`) и
  сносятся **отложенным `LeaveScope`, который срабатывает до запуска
  `Exec`** (defer стоит внутри `prepareNodeExecution`, `track.go:514`) —
  чтения узла удаются лишь потому, что незаблокированный `getData` уходит к
  корню после исчезновения пути узла.
- Данные выполнения живут в **мутабельных полях узлов**: `activity.dataPath`
  (`activity.go:29`, запись `task.go:129`), `Event.dataPath`
  (`event.go:125`, запись `start.go:135` / `end.go:158`, чтение
  `event.go:430`), `ExclusiveGateway.scope` (`exclusive.go:18`, запись в
  `Exec` `exclusive.go:77`, чтение в `Find` `exclusive.go:163-171`),
  `UserTask.resChan` (`user_task.go:42`, запись/чтение только внутри `Exec`,
  `user_task.go:194/198`). ADR-009 сделал их per-instance; они по-прежнему
  per-*node* там, где обязаны быть per-*execution* — циклы, Parallel-форк с
  пересечением одного узла двумя треками ломают их по построению.
- **Разделяемые экземпляры параметров мутируются на месте**:
  `IoSpec.Parameters()` возвращает живые указатели `*Parameter`
  (`io_spec.go:109-123`); `task.LoadData` пишет в них (`task.go:106,114`),
  `task.updateOutputs` пишет значения и переключает состояние
  (`task.go:222-231`), `throwEvent.LoadData` делает то же с разделяемыми
  `dataInputs` (`event.go:381-392`), а `ServiceTask` мутирует item'ы
  сообщений своей операции (`service_task.go:183-184`, `:209-213`). Один
  набор экземпляров на узел, разделяемый между выполнениями.
- `catchEvent.UploadData(ctx)` (`event.go:263`) имеет **неверную сигнатуру**
  для `scope.NodeDataProducer` (`UploadData(ctx, s Scope)`), поэтому
  проверка роли производителя у трека (`track.go:633`) никогда не совпадает
  с catch-событием — выгрузка данных catch-стороны сегодня мёртвый код.

**Всё — это Instance:** `renv.RuntimeEnvironment` встраивает `scope.Scope`
(`renv.go:19`), трек передаёт `t.instance` в каждый `Exec` узла
(`track.go:529`) и в `UploadData` (`track.go:638`) — та самая связка
инстанс-как-scope из находки §2.3 аудита про god object.

**Runtime-переменные:** `GetData` перехватывает зарезервированный путь
`/<process>/RUNTIME` (`instance.go:751-753`) и синтезирует параметры
`STARTED_AT`, `STATE`, `TRACKS_CNT` по требованию (`getRuntimeVar`,
`instance.go:574-618`). Сегодня их читает только `TestMonitoring`
(`instance_test.go:47-75`). Решение владельца (ревью, 2026-06-12): фича
**остаётся**, обслуживается data plane'ом как зарезервированное read-only
поддерево.

### 1.2 Почему

ADR-010 v.1 решает модель; Parallel-шлюз (ADR-005) сделал конкурентные треки
рутиной, так что каждый дефект выше лежит на горячем пути выполнения. По
триажу аудита находка §1.2 устраняется здесь — структурно, а не правкой
расстановки блокировок.

## 2. Цели и область

### 2.1 Цели (в области)

- **G1.** Выделенный компонент data plane владеет деревом контейнерных
  scope'ов; каждая операция атомарна под одной блокировкой (ADR-010 §2.2).
- **G2.** Каждое выполнение узла работает на frame'е выполнения с ключом
  (track, node), с per-frame экземплярами параметров/свойств, атомарным
  батч-коммитом при успехе, отбрасыванием при сбое (ADR-010 §2.3).
- **G3.** Узлы держат только иммутабельные определения данных +
  lifetime-состояние ADR-009: четыре поля данных выполнения и
  `NodeDataLoader`/`RegisterData` удаляются (ADR-010 §2.4).
- **G4.** Трек вручает каждому узлу per-execution окружение; `renv` перестаёт
  встраивать `scope.Scope`; форма `Exec(ctx, renv.RuntimeEnvironment)` не
  меняется (ADR-010 §2.4).
- **G5.** `-race`-чистота для конкурентных треков, коммитящих в один scope, и
  для двух треков, пересекающих один узел; код §1.2 аудита физически удалён.
- **G6.** RUNTIME-переменные выживают как зарезервированное read-only
  поддерево data plane (решение владельца, §1.1).
- **G7.** Рабочий пример упражняет персистентную модель данных (сегодня —
  ни один); ADR-010 переходит в Принято с RU-twin'ами; roadmap/SAD/cross-doc
  пины синхронизированы.

### 2.2 Не-цели (отложено)

- Модельная семантика data flow — порядок выбора InputSet, гейтинг
  доступности («unavailable»), семантика `DataState`, структурная ревизия
  `IoSpec`/`InputSet`: планируемый **data-flow ADR**. Текущее поведение
  вычисления сохраняется как есть, лишь перенацеливается на frame'ы.
- `DataStore`, долговечная персистентность, регидрация — будущий
  Persistence & State ADR.
- Дочерние scope'ы подпроцессов — дерево и `OpenScope`/`CloseScope` для них
  существуют, но подпроцессы здесь не приземляются (roadmap WS-C4).
- Подписки на изменения scope'а / слушатели — observability ADR.
- Дефекты событийной подсистемы (аудит 1.3/1.4/1.5) — FIX-прогон.

## 3. Требования

### 3.1 Функциональные

| # | Требование |
|---|---|
| FR-1 | **`internal/scope` становится data plane'ом.** *Интерфейс* `Scope` (`scope.go:31-64`) и `NodeDataLoader` (`scope.go:69-80`) удаляются; `Scope` возрождается как **конкретная структура**, владеющая деревом контейнерных scope'ов: `map[DataPath]map[string]data.Data` + один `sync.Mutex`. Операции — `GetData(from DataPath, name string)`, `GetDataByID(from DataPath, id string)` (обе идут вверх к корню), `Commit(at DataPath, dd ...data.Data)` (атомарный батч, единственный писатель), `OpenScope(DataPath)` / `CloseScope(DataPath)` (дочерние контейнеры; сейчас для корня, позже для подпроцессов), `Root()` — каждая целиком под мьютексом; никакая составная операция не разорвана между захватами. Все публичные методы валидируют каждый параметр (nil-данные, пустые имена, невалидные пути). |
| FR-2 | **`scope.Frame`** — per-execution рабочий набор, создаваемый треком на каждое выполнение узла, идентичность (trackID, nodeID): per-frame **экземпляры входных/выходных `*data.Parameter`**, построенные из определений `IoSpec` узла через `NewParameter(name, iae.Clone())` (`io_spec_obj.go:52`, `item.go:321`), per-frame **экземпляры свойств** (свойства activity/события — видимы только этому выполнению, по стандарту), и поиск, разрешающий frame-first, затем обход контейнеров. `Frame.Commit()` сбрасывает выходы в контейнерный scope одним батчем `Scope.Commit`; `Frame.Discard()` отбрасывает всё (путь сбоя — контейнерный scope не наблюдает ничего). |
| FR-3 | **Per-execution окружение.** `renv.RuntimeEnvironment` (`renv.go:16-28`) теряет встроенный `scope.Scope` и вместо него открывает frame-backed доступ к данным (`GetData`/`GetDataByID` с разрешением frame → контейнер; реализует `data.Source` для вычисления выражений). | FR-4 | **Переработка хуков трека** (`track.go:494-546`): `prepareNodeExecution` создаёт frame и выполняет роль потребителя (загрузка входов во frame); frame живёт через `Exec`; `finalizeNodeExecution` выполняет роль производителя и `Frame.Commit()`; пути сбоя/прерывания вызывают `Frame.Discard()`. Дефект Leave-до-Exec исчезает вместе с самими `ExtendScope`/`LeaveScope`. |
| FR-5 | **Миграция узлов — activities.** `task`: удалить `dataPath` + `RegisterData` (`task.go:128-151`); `LoadData`/`UploadData` перенацеливаются на экземпляры параметров frame'а (сегодняшняя мутация на месте разделяемых указателей `IoSpec` в `task.go:106-114`, `:222-231` прекращается). `ServiceTask`: per-execution item'ы сообщений операции (экземпляры во frame'е; `service_task.go:183`, `:209-213` перенацеливаются). `UserTask`: `resChan` становится локальной переменной `Exec` (сегодня запись/чтение только внутри `Exec`, `user_task.go:194/198`); данные результата идут в выходы frame'а, коммитимые треком — путь self-deadlock'а `re.AddData` (`user_task.go:202`) удаляется вместе с самим `AddData`. |
| FR-6 | **Миграция узлов — события.** `Event.dataPath` + `RegisterData` удаляются (`event.go:124-125`, `start.go:134-138`, `end.go:157-161`); `throwEvent.LoadData` заполняет frame-экземпляры входов (сегодня `event.go:381-392` мутирует разделяемые карты); чтение scope'а в `emitEvent` (`event.go:430`) разрешается через per-execution окружение; `catchEvent.UploadData` получает сигнатуру роли производителя и **оживает** (сегодня мёртв: неверная сигнатура, `event.go:263`), коммитя выходы catch через frame. |

## 4. Дизайн и план реализации

### 4.1 Data plane (`internal/scope`)

```go
// Scope is the per-instance data plane: the container-scope tree and the
// single authority for persistent process data (ADR-010 §2.2).
type Scope struct {
    m      sync.Mutex
    root   DataPath
    scopes map[DataPath]map[string]data.Data
    rt     RuntimeVarsSupplier // reserved /RUNTIME subtree (FR-9)
}

func New(root DataPath, rt RuntimeVarsSupplier) (*Scope, error)
func (s *Scope) Root() DataPath
func (s *Scope) GetData(from DataPath, name string) (data.Data, error)
func (s *Scope) GetDataByID(from DataPath, id string) (data.Data, error)
func (s *Scope) Commit(at DataPath, dd ...data.Data) error
func (s *Scope) OpenScope(p DataPath) error
func (s *Scope) CloseScope(p DataPath) error
```

`DataPath` и его методы (`datapath.go`) живут без изменений — ограничены
адресацией контейнеров. Возрождение интерфейс→структура сохраняет пакет и
имя (концепция ведёт, имена следуют); узлы структуру не видят — они видят
per-execution окружение.

### 4.2 Frame (`internal/scope`)

```go
// Frame is the working set of one node execution by one track
// (ADR-010 §2.3): per-execution parameter and property instances,
// frame-first lookup, all-or-nothing commit.
type Frame struct {
    trackID string
    nodeID  string
    at      DataPath           // the containing container scope
    inputs  []*data.Parameter  // per-frame instances (NewParameter + IAE.Clone)
    outputs []*data.Parameter
    props   []data.Data        // node property instances
    scope   *Scope
}

func NewFrame(trackID, nodeID string, at DataPath, s *Scope) (*Frame, error)
func (f *Frame) GetData(name string) (data.Data, error)   // frame → container walk
func (f *Frame) GetDataByID(id string) (data.Data, error)
func (f *Frame) Put(dd ...data.Data) error                // node-produced values
func (f *Frame) Commit() error                            // one Scope.Commit batch
func (f *Frame) Discard()
```

Носители, не пересекающие границу prepare/execute/finalize (канал ответа
`UserTask`, источник вычисления шлюза), становятся локальными переменными
`Exec` — per-execution по построению, ничего на узле. Frame несёт то, что
пересекает границы стадий: экземпляры I/O-параметров, свойства,
per-execution item'ы сообщений операции `ServiceTask`.

### 4.3 Per-execution окружение

`internal/renv.RuntimeEnvironment` сохраняет `engrenv.EngineRuntime`,
`InstanceID()`, `EventProducer()`, `RenderRegistrator()` и заменяет
встроенный `scope.Scope` frame-backed поверхностью (`GetData`,
`GetDataByID`; он реализует `data.Source`, так что вычисление выражений
читает через него — заменяя паттерн захваченного scope'а в
`ExclusiveGateway.Find`). Трек строит по одному на выполнение: `сервисы
инстанса + frame`. Моки перегенерируются; каждое тело `Exec` узла, звавшее
`re.GetDataByID(st.dataPath,…)`, теперь зовёт `re.GetDataByID(…)` —
аргумент-путь исчезает вместе с хранимым узлом `DataPath`.

### 4.4 Роли потребителя/производителя

`scope.NodeDataConsumer` / `scope.NodeDataProducer` выживают с
frame-ориентированными сигнатурами (точная форма пиннится на код-ревью M3;
контракт таков: потребитель заполняет входы frame'а из своих ассоциаций,
производитель выдаёт выходы для коммита треком). `catchEvent.UploadData`
выравнивается под сигнатуру производителя и становится достижимым (FR-6).

### 4.5 Майлстоуны (каждый = один коммит, независимо CI-зелёный)

> **Замечание о порядке.** Замена встроенного `scope.Scope` в `renv`
> прокатывается через все узлы разом — поэтому новые типы приземляются
> первыми (невподключёнными, M2…M3), а переключение — один атомарный
> майлстоун (M4), зеркаля M1 SRD-006 с его разовой сменой интерфейса.

- **M2 — data plane.** Конкретный `Scope` + `RuntimeVarsSupplier` (FR-1,
  FR-9) с полными unit-тестами: атомарность, разрешение вверх, батч-коммит,
  open/close, read-only зарезервированного поддерева, валидация параметров и
  конкурентный `-race`-тест (NFR-1/2). Старый код не тронут; оба
  сосуществуют.
- **M3 — frame.** `scope.Frame` (FR-2) + тесты: per-frame экземпляры
  (мутация параметра frame'а не трогает определения `IoSpec` узла и другие
  frame'ы), frame-first поиск, батч коммита, отбрасывание. Невподключён.
- **M3a — фикс конкурентности генератора id** (обнаружен на M2, добавлен с
  одобрения владельца): генератор id в `foundation` гонял
  несинхронизированный `math/rand.Rand` и лениво инициализировал свою
  пакетную переменную гоночным check-then-assign — конкурентное
  конструирование элементов модели было гонкой данных. Исправлено
  (горутинобезопасный top-level `rand`, жадная инициализация, `RWMutex` на
  смене/чтении генератора) с `-race`-регрессионным тестом. Жёсткое
  предусловие M4: frame'ы инстанцируют экземпляры параметров из конкурентных
  треков, так что каждое пересечение узла генерирует id конкурентно.
- **M4 — переключение.** `renv` худеет (FR-3); переработка хуков трека
  (FR-4); миграция узлов (FR-5/6/7); Instance снимает scope (FR-8); удаления

## 6. Риски и регрессии

- **Радиус поражения M4.** Похудение `renv` трогает каждый вид узла, трек,
  Instance, моки и тесты одним коммитом. Митигировано: M2/M3 приземляют
  новый мир полностью протестированным заранее; M4 — механическое
  перенацеливание на стабильные контракты; per-kind тесты V5 пишутся
  *вместе с* M4.
- **Конкурентное конструирование за пределами data plane.** Инстанцирование
  frame'а строит элементы модели (параметры, item-aware элементы) из
  конкурентных горутин треков — каждый разделяемый глобал на пути
  конструирования — кандидат в гонку. Генератор id был подтверждённым
  случаем (исправлен в M3a); `-race`-тест пересечения V3 — постоянный страж
  остального пути.
- **Точность клонирования экземпляров параметров.**
  `ItemAwareElement.Clone()` (`item.go:321`) обязан давать
  value-независимые копии для каждого вида `values.Value`, который может
  нести frame; неглубокое место молча вернуло бы разделение состояния. V3
  явно утверждает независимость определение-vs-frame.
- **Контракт expression-движка.** Условия `ExclusiveGateway` читают
  переменные через `data.Source.Find` (`exclusive.go:142,159-171`);
  окружение обязано сохранить семантику разрешения `Find` (поиск имени от
  позиции шлюза), иначе условия молча перестанут видеть данные. Покрыто
  существующими тестами шлюза, перенацеленными в M4.
- **Сообщения операции ServiceTask.** Per-execution экземпляры сообщений
  обязаны сохранить контракт `Operation.Run` (`operation.go:202`); путь
  `gooper`-функторов (оба примера) — регрессионная канарейка (V7).
- **Поведенческая заметка (намеренная):** значения, закоммиченные узлом,
  становятся видимы *другим* трекам только при коммите frame'а — сегодняшняя
  видимость разделяемой карты посреди выполнения сужается до семантики
  копирования/коммита стандарта. Ни один текущий тест или пример не зависит
  от видимости посреди выполнения (проверено: примеры данных не несут; тесты
  мокают scope).

## 7. Сводка реализации

Приземлено на ветке `feat/process-data-model` (от `master`).

**Коммиты майлстоунов**

- `d30dd2c` — M2: data plane (структура `Scope` в `internal/scope` —
  приземлена под переходным экспортированным именем `Plane`, переименована в
  M4 — с атомарностью целых операций, батч-`Commit`, зарезервированным
  поддеревом RUNTIME).
- `62dfc76` — M3: `scope.Frame` (per-frame экземпляры через `IAE.Clone`,
  frame-first поиск, коммит всё-или-ничего, отбрасывание при сбое).
- `389f075` — M3a: гонко-безопасная генерация id (`foundation` — top-level
  `math/rand`, жадная инициализация, `RWMutex` на смене генератора).
- `fbfc018` — M4: атомарное переключение (старый интерфейс `Scope` +
  `NodeDataLoader` удалены; `renv` похудел до per-execution поверхности;
  `execEnv`; жизненный цикл frame'а в треке; все виды узлов мигрированы;
  Instance владеет plane'ом и поставляет RUNTIME-переменные; моки
  перегенерированы).
- `6c86620` — M5: `examples/process-data` (FR-11).

**Детали контракта, закреплённые при реализации** (по §4.4 точные сигнатуры
ролей были концерном M3/M4):

- `NodeDataConsumer.LoadData(ctx, *scope.Frame)` /
  `NodeDataProducer.UploadData(ctx, *scope.Frame)`.
- Порядок разрешения frame'а: входы → свойства → puts → обход контейнеров.
  **Экземпляры выходов никогда не разрешаются** — они цели записи;
  разрешение позволило бы ещё-не-заполненному выходу затенить данные,
  которыми его должны заполнить на стадии производителя (`updateOutputs`
  тянет значение по ItemDefinition id самого выхода).
- `DataInput`, заполненный своей ассоциацией, переводит свой frame-экземпляр
  в Ready (BPMN §10.4.2) — раньше флип не делал никто, так что
  реальный путь входов не мог завершиться (он существовал только за моками
  scope'а).
- `ServiceTask` запускает операцию на per-execution клоне; результат попадает
  во frame как put с ключом-id item'а исходящего сообщения. Прежний поток
  сообщений был ИНВЕРТИРОВАН (выход копировался в сообщение операции;
  результат запуска никогда не достигал выхода) — исправлено; отсутствие
  декларации выхода больше не ошибка времени выполнения (валидация модели —
  забота data-flow ADR).
- `catchEvent.UploadData` ожил (старая сигнатура никогда не удовлетворяла
  роль производителя); у ассоциаций выходов всё ещё нет API связывания
  (работа message-корреляции, WS-C3).

**Обнаружено и исправлено сверх плана**

- M3a (вставка с одобрения владельца): генератор id был тройной гонкой
  данных — несинхронизированный `rand.Rand`, гоночная ленивая инициализация,
  несинхронизированный сеттер.
- Унаследованный self-deadlock `Instance.AddData` и дефект
  Leave-scope-до-Exec умерли вместе со своим кодом (§1.1).
