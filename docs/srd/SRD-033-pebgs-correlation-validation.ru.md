# SRD-033 — Parallel-start event gateway: валидация корреляции и рантайм-страж ключа

| Поле | Значение |
|---|---|
| Статус | Принято |
| Версия | v.1 |
| Дата | 2026-07-02 |
| Владелец | Руслан Габитов |
| Реализует | [ADR-005 v.4 §2.12.5(7) Gateways & Joins](../design/ADR-005-gateways-and-joins.ru.md) |

Реализует то единственное правило валидации из ADR-005 v.4 §2.12.5, которое так и
не приземлилось в коде — правило 7, *«арки Parallel-start должны нести
корреляцию»*, — закрывая находку аудита **AB-001** (безключевой стартовый шлюз
`ParallelEvents` создаёт по инстансу на арку): отвергнуть неверную конфигурацию
на регистрации и подстраховать рантайм-путь, где конформная модель всё же
встречает неконформное сообщение.

## 1. Предпосылки (проверено по коду)

### 1.1 Дефект (AB-001)

Инстанцирующий Event-Based gateway в режиме `ParallelEvents` (PEBGS) должен
породить **один** инстанс на разговор: сообщение первой арки создаёт его,
сообщения остальных арок присоединяются к нему, а завершение ждёт все арки
(BPMN §13.2). Решение create-or-route ключуется на выведенном корреляционном
ключе сообщения:

- `instanceStarter.deriveKey` (`pkg/thresher/instance_starter.go:71`) возвращает
  `""`, когда у стартера нет `CorrelationKey` (`if s.corrKey == nil`) — и когда
  ключ нельзя populate из полезной нагрузки (`DeriveKey` → `ok == false`).
- `Thresher.resolveAndLaunch` трактует пустой ключ как «без дедупликации» —
  **всегда создаёт новый инстанс**.

Так безключевой PEBGS плодит по инстансу на сообщение арки: N арок → N
подвешенных инстансов, каждый вечно ждёт арки-соседи (чьи сообщения ушли в
соседние инстансы). `EventBasedGateway.Validate` пропускает безключевой шлюз —
у `validateStartGate` нет проверки корреляции.

### 1.2 Решение уже принято — этот SRD его приземляет

[ADR-005 v.4 §2.12.5](../design/ADR-005-gateways-and-joins.ru.md) решает восемь
правил валидации шлюза на регистрации. Код реализует правила 1–6 и 8; **правило 7
описано, но не реализовано**: *«арки Parallel-start должны нести корреляцию,
чтобы последующие события маршрутизировались в созданный инстанс»*. Нормативная
основа — **BPMN §10.6.6** (стр. 298, formal/13-12-09, сверено с PDF): «сообщения,
запускающие события конфигурации шлюза, ДОЛЖНЫ разделять одну и ту же
корреляционную информацию». Клауза присутствует в вендоренном экстракте
`docs/bpmn-spec/state-machines/process-lifecycle.md`.

### 1.3 Что «разделять корреляцию» значит механически

`msgflow.DeriveKey` собирает ключ из свойств `CorrelationKey`; для каждого
свойства ищет retrieval-выражение **для этого сообщения** —
`retrievalExprFor(prop, msg)` матчит `MessageRef.Name() == msg.Name()` — и
провал (`ok=false`), когда у свойства нет выражения для сообщения. Значит арка
«несёт корреляцию» ровно когда **каждое свойство ключа шлюза объявляет
retrieval-выражение для сообщения этой арки** — чисто структурное условие,
проверяемое на регистрации без вычисления выражений.

### 1.4 Семантика «одно сообщение на арку» — уже верна, здесь закрепляется тестом

Каждая оставшаяся арка рождённого PEBGS-инстанса засеивается как припаркованный
трек на узле арки (`Instance.seedParallelStart`), который регистрирует
**одноразовый** in-instance waiter (`EventHub.RegisterEvent`; хаб удаляет
сработавший waiter — ADR-006 v.1 §2.5). Значит арка потребляет ровно одно
сообщение, а её подписка умирает со срабатыванием; дубликат того же ключа на уже
сработавшей арке доходит только до персистентного стартера, чья резервация ключа
делает его no-op «присоединиться». Механику менять не нужно; FR-4 закрепляет
гарантию регресс-тестом.

## 2. Требования

### Функциональные

- **FR-1 — безключевой PEBGS отвергается на регистрации.** Инстанцирующий шлюз
  `ParallelEvents` без `CorrelationKey` проваливает `EventBasedGateway.Validate`
  (и `Process.Validate` / `snapshot.New`) с самоидентифицирующей ошибкой,
  называющей шлюз.
- **FR-2 — покрытие ключа каждой аркой отвергается на регистрации.** Для
  ключевого PEBGS сообщение каждой арки должно покрывать ключ: каждое
  `CorrelationProperty` ключа объявляет retrieval-выражение с `MessageRef` этой
  арки. Нарушение проваливает валидацию, называя шлюз, арку и непокрытые
  свойства.
- **FR-3 — рантайм-страж невыводимого ключа (только стартер PEBGS).** Когда
  персистентный стартер `ParallelEvents`-шлюза выводит пустой ключ в рантайме
  (нагрузка не populate ключ — `DeriveKey ok=false`), стартер **не
  инстанцирует**: сообщение потребляется с Warn-логом (неконформный ввод по
  §10.6.6). Семантики Exclusive-start и message-start не затрагиваются.
- **FR-4 — одно сообщение на арку (регресс-закрепление).** После рождения
  PEBGS-инстанса и срабатывания одной из оставшихся арок второе сообщение с тем
  же корреляционным ключом для этой арки не пересрабатывает её и не создаёт ещё
  один инстанс.

### Нефункциональные

- **NFR-1 — никаких значений нагрузки в логах.** Warn FR-3 логирует id процесса,
  id шлюза и имя сообщения — никогда содержимое нагрузки (прецедент NFR-1
  `DeriveKey`).
- **NFR-2 — ошибки валидации самоидентифицируемы** (id шлюза, id арки, имена
  свойств), как и существующие проверки §2.12.5.
- **NFR-3 — никаких изменений поведения для валидных моделей.** Ключевой,
  полностью покрытый PEBGS и любая не-PEBGS конфигурация валидируются и
  исполняются как прежде.

## 3. Модели

### 3.1 Статический помощник покрытия `msgflow` (`pkg/model/msgflow/correlation.go`)

Небольшой экспортируемый помощник рядом с `DeriveKey`, переиспользующий
`retrievalExprFor`:

```go
func MissingKeyProperties(
	key *bpmncommon.CorrelationKey,
	msg *bpmncommon.Message,
) []string
```

Возвращает свойства ключа без retrieval-выражения для `msg` (пустой результат =
покрыт). Ребро `gateways → msgflow` новое и без циклов.

### 3.2 Валидация шлюза (`pkg/model/gateways/event_based.go`)

`validateStartGate` получает проверки правила 7, активные только при
`instantiate && gwType == ParallelEvents`: `g.corrKey == nil` → ошибка (FR-1);
иначе для каждой арки резолвится её сообщение (Receive-Task — через собственный
`Message()`, message-catch — через `*events.MessageEventDefinition`) и
отвергается при непустом `msgflow.MissingKeyProperties` (FR-2).

### 3.3 Рантайм-страж стартера (`pkg/thresher/instance_starter.go`)

`ProcessEvent` получает ветку между `deriveKey` и `resolveAndLaunch`: если
выведенный ключ пуст **и** `startNode` — Parallel-start-шлюз
(`interface{ ParallelStart() bool }`), Warn-лог и возврат без инстанцирования
(FR-3).

## 4. Анализ

### 4.1 Отвергать на валидации, а не неявный ключ по id шлюза (решено)

Ранний набросок AB-001 взвешивал ключ по id шлюза при отсутствии
`CorrelationKey`. Отвергнуто: §10.6.6 **предписывает** разделяемую корреляцию,
так что безключевая модель неконформна — «заставить работать» через
изобретённый движком ключ замаскировало бы ошибку моделирования и породило бы
singleton-семантику, которой стандарт не определяет. Fail-fast на регистрации
соответствует приземлённым правилам §2.12.5 и дисциплине «валидировать весь
ввод».

### 4.2 Статическая проверка + рантайм-страж, а не одно из двух (решено)

Статика (FR-1/FR-2) ловит ошибки модели на регистрации, но не видит нагрузок;
страж (FR-3) ловит конформные модели с неконформными сообщениями (незаполненное
опциональное поле), которые иначе снова открыли бы баг через ветку создания на
пустом ключе. Оба нужны; страж ограничен стартером PEBGS.

### 4.3 Резолв сообщения арки через существующий импорт `events` (решено)

`gateways` уже импортирует `events`; message-catch-арка читается напрямую из её
`*events.MessageEventDefinition`, Receive-Task — через свой `Message()`. Единственное
новое ребро — `gateways → msgflow` (§3.1), без циклов.

### 4.4 Одноразовым аркам механика не нужна (решено)

§1.4: одноразовые in-instance waiters уже реализуют «событие происходит один
раз»; резервация ключа стартером уже поглощает дубликаты. FR-4 добавляет лишь
тест.

### 4.5 Что остаётся без изменений (решено)

Ветка создания на пустом ключе (легитимна для безключевых message-start и
signal-start), `DeriveKey`, `seedParallelStart`, жизненный цикл waiter'а и все
правила §2.12.5 1–6/8.

## 5. API / поверхность контракта

- **Новое экспортируемое:** `msgflow.MissingKeyProperties`.
- **Изменённый контракт:** `EventBasedGateway.Validate` отвергает две ранее
  принимавшиеся (но никогда не исполнимые до завершения) конфигурации:
  безключевой PEBGS и ключевой PEBGS с непокрытой аркой.
- **Без изменения сигнатур**; страж FR-3 внутренний.
