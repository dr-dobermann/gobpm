> EN-оригинал — канонический: [SRD-006-per-instance-node-graph.md](SRD-006-per-instance-node-graph.md). Этот файл — его перевод (twin).

Этот SRD приземляет [ADR-009](../design/ADR-009-per-instance-node-graph.md): каждый Instance **клонирует** неизменяемый граф узлов процесса в свой приватный граф, и узлы держат своё **состояние рантайма на каждый instance** напрямую. Это убирает гонку на разделяемом узле и даёт синхронизирующим шлюзам / таймерам / корреляции дом на узле. Это фундамент, на котором возобновляется работа над Parallel-шлюзом (ADR-005).

## 1. Предпосылки и мотивация

### 1.1 Текущее состояние (сверено с кодом)

- **snapshot** разделяет указатели на узлы процесса — `snapshot.go:51-52` `s.Nodes[n.ID()] = n` (без копии) — и один snapshot на процесс переиспользуется для **каждого** instance (`thresher` `StartProcess` → `launchInstance(s)`). Так что все instance процесса разделяют **одни и те же объекты узлов**.
- `instance.New` держит разделяемый snapshot (`instance.go:151` `s: s`); `createTracks` итерирует `inst.s.Nodes` (`instance.go:427`), и `newTrack(n, …)` исполняет **разделяемый** узел; форки разрешают следующий узел через **обратную ссылку flow** `f.Target().Node()` (`instance.go:342`, `track.go:527`), а не через `s.Nodes`.
- **Исполнение мутирует поля на разделяемом узле** (гонка, помеченная ADR-001 §4.7 / ADR-005): `Event.dataPath` (`event.go:125`; пишется `RegisterData` — Start `start.go:125`, End `end.go:149` — через `ExtendScope`→`RegisterData` `instance.go:777`), `activity.dataPath` (`activity.go:29`; `task.go:119`, покрывает Service+User), `UserTask.resChan` (`user_task.go:43/175`), `ExclusiveGateway.scope` (`exclusive.go:18/59`). Под `-race` (включён по умолчанию — `Makefile:117/119`) два конкурентных instance над одним узлом дали бы гонку.
- Более глубокий разделятель: `ServiceTask.operation` (`*service.Operation`, `service_task.go:36`) имеет свои Items сообщения, мутируемые при исполнении (`service_task.go:170-198`) — разделение его между instance'ами заново вносит гонку даже после изоляции полей выше.
- Прецедент машинерии клонирования существует для *данных*: `data.ItemAwareElement.Clone()` (`item.go:321`) — тот же ID, независимое значение — и `flow.EventDefCloner.CloneEventDefinition` (`events.go:65`, используется в `throwEvent.emitEvent` `event.go:410`). Никакого `Clone()` на узле/flow не существует.

### 1.2 Почему

Состояние рантайма на каждый instance (ADR-009) — фундамент для синхронизирующих join'ов (ADR-005), таймеров и корреляции, и оно устраняет гонку на разделяемом узле. Без него ни у чего из этого нет чистого дома, и гонка остаётся.

## 2. Цели и область

### 2.1 Цели (в области)

- **G1.** Каждый Instance владеет **клонированным** графом узлов; snapshot остаётся разделяемым неизменяемым шаблоном.
- **G2.** Каждый конкретный тип узла имеет `Clone()` (разделяет неизменяемый конфиг по ссылке, свежее состояние рантайма, пустые flow); топология клонированного графа перепрошита, так что `Source()/Target()` разрешаются в **клонированные** узлы.
- **G3.** Мутируемые при исполнении поля становятся **на каждый instance**: `Event.dataPath`, `activity.dataPath`, `UserTask.resChan`, `ExclusiveGateway.scope` и мутируемое при исполнении состояние сообщения operation у `ServiceTask`.
- **G4.** Гонка на разделяемом узле ушла — `-race` чисто для ≥2 конкурентных instance одного процесса.
- **G5.** Согласовать **ADR-001 §4.7** (владение состоянием рантайма решено; долговечная персистентность всё ещё отложена) и обновить **roadmap** (добавить ADR-009).

### 2.2 Не-цели (отложено)

- Долговечная персистентность / сериализация / регидрация / восстановление после рестарта — будущий Persistence & State ADR (ADR-009 §3).
- Сам join шлюза (ADR-005 / SRD-005, возобновляется после этого).
- Новые типы узлов (Script/Send/Receive/BoundaryEvent) — только 5 существующих исполняемых типов получают `Clone()`.

## 3. Требования

### 3.1 Функциональные

| # | Требование |
|---|---|
| FR-1 | `snapshot.(*Snapshot).Clone()` возвращает **`*Snapshot` на каждый instance** с клонированными `Nodes` + перевязанными `Flows` (неизменяемый заголовок — `ProcessID`/`ProcessName`/`Properties` — разделяется по ссылке). `instance.New` вызывает его и держит **свой собственный клонированный snapshot** вместо разделяемого; snapshot thresher'а остаётся разделяемым **шаблоном** и никогда не мутируется. `createTracks` уже итерирует `inst.s.Nodes`, поэтому он теперь итерирует **клонированные** узлы без дальнейших изменений. |
| FR-2 | Каждый из **5 конкретных типов узлов** — `events.StartEvent`, `events.EndEvent`, `activities.ServiceTask`, `activities.UserTask`, `gateways.ExclusiveGateway` — реализует `Clone() flow.Node`, воспроизводя поля встроенных родителей (`Event`/`catchEvent`/`throwEvent`, `activity`/`task`, `Gateway`, `flow.BaseNode`): **неизменяемый конфиг разделяется по ссылке**, **состояние рантайма свежее**, **flow пустые** (перепрошиваются FR-4). |
| FR-3 | Мутируемые в рантайме поля становятся **на каждый instance** (свежие/нулевые при клонировании): `Event.dataPath` (`event.go:125`), `activity.dataPath` (`activity.go:29`), `UserTask.resChan` (`user_task.go:43`), `ExclusiveGateway.scope` (`exclusive.go:18`). |
| FR-4 | **Хелпер клонирования из пакета `flow`** перестраивает топологию между клонированными узлами: для каждого исходного `SequenceFlow` создаёт клонированный flow, который **сохраняет id и условие flow**, ставит `source`/`target` на **клонированные** концы и встраивает его в карты flow обоих клонированных узлов. Он **не должен** регенерировать id и **не должен** пытаться вставку в контейнер (клоны не несут разделяемого контейнера). (`flow.Link`→`connect` `sequenceflow.go:157` — референс прошивки; `flows` неэкспортируемо, поэтому перепрошивка живёт в пакете `flow`.) |
| FR-5 | `ServiceTask.Clone()` даёт **на каждый instance** operation, так что мутируемое при исполнении состояние сообщения (`service_task.go:170-198`) не разделяется между instance'ами — клонировать `*service.Operation` (разделить его неизменяемое определение, свежие переносчики сообщений). |
| FR-6 | Клонированные узлы не несут **межинстансовой обратной ссылки** — `flow.BaseElement.container` (`element.go:95`) очищается/локален для instance, так что два instance полностью независимы и `Link`-подобные вставки в контейнер не срабатывают на клонах. |
| FR-7 | **Задокументировать** границу кеинга подписки на события: EventHub кеит ожидающих по `eDef.ID()` **в одиночку** (`eventhub.go:118/145`), а `Event.clone()` разделяет определения событий по ссылке (M1), поэтому клонированные узлы event-start/catch двух instance'ов несут один и тот же id определения и разделили бы ожидающего. Изоляция **подписки на события** на каждый instance (кеинг по `(processor, defID)` / подписки на каждый instance) — это **семантика доставки событий, принадлежащая ADR-006**, а не решение о состоянии узла на каждый instance этого SRD — **отложена в ADR-006**. Гонка состояния узла на каждый instance, которую устраняет этот SRD, живёт на **простом** flow (без ожидающих событий); FR-1…6 и race-тест V4 покрывают её на простом процессе. |

### 3.2 Нефункциональные

| # | Требование |
|---|---|
| NFR-1 | `-race` чисто для ≥2 конкурентных instance одного процесса, исполняющихся над ранее-разделяемыми узлами (гонка §3 устранена). |
| NFR-2 | Старт на каждый instance — O(nodes + flows); неизменяемый конфиг (определения, шаблоны operation, условия) разделяется по ссылке — память остаётся низкой. |
| NFR-3 | Никакого изменения поведения для исполнения одиночного instance; все существующие тесты + все три примера по-прежнему проходят. |
| NFR-4 | diff-coverage ≥80 % (цель 100 %) на затронутых файлах. |

## 4. Дизайн и план реализации

### 4.1 Механизм клонирования

- **Per-type `Clone()`** (в собственном пакете каждого узла, чтобы он мог трогать свои поля): копировать поля конфига **по ссылке** (определения событий, шаблон operation, условия, направление, default-flow, id, имена), аллоцировать **свежие** поля состояния рантайма (FR-3) и начинать с **пустых** `BaseNode.flows`. Встроенные родители выставляют внутренние хелперы клонирования (`Event.clone`, `activity.clone`, `task.clone`, `Gateway.clone`, `BaseNode.cloneShell`).
- **`Snapshot.Clone()` зеркалит `snapshot.New`.** `snapshot.New` строит шаблон двумя независимыми циклами: узлы (`snapshot.go:51-52`) и flow (`snapshot.go:87-89`) в id-кеированные карты. `Clone()` переиспользует ту же самую форму на построенном snapshot'е: итерировать `s.Nodes` → `n.Clone()` в карту `id→clone`; итерировать `s.Flows` → пересоздать каждое ребро **между клонами** (сохранить id + условие, без вставки в контейнер); копировать неизменяемый заголовок (`ProcessID`/`ProcessName`/`Properties`) по ссылке. Поскольку snapshot уже перечисляет каждый узел и flow, **обход графа не нужен** — пересоздание ребра — это маленький хелпер пакета `flow` (`flows` неэкспортируемо). Возвращает новый `*Snapshot`.
- **Hook:** `instance.New` вызывает `s.Clone()` один раз на старте и сохраняет возвращённый `*Snapshot` на каждый instance как `inst.s`; snapshot thresher'а остаётся разделяемым шаблоном. `createTracks` (`instance.go:427`) и форки уже читают `inst.s.Nodes` / граф flow, поэтому прозрачно используют клон.

### 4.2 Миграция полей состояния

Четыре поля в FR-3 уже живут на узле; единственное изменение в том, что они теперь **на каждый instance**, потому что узел на каждый instance — клон обнуляет их, исполнение пишет их на собственном узле instance. Никаких изменений сигнатур; гонка §3 ушла, потому что записи нацелены на приватный узел.

### 4.3 `service.Operation` на каждый instance

`ServiceTask.Clone()` клонирует operation (разделить неизменяемое определение operation; свежие переносчики сообщений), так что `loadInputMessage`/`uploadOutputMessage` мутируют состояние на каждый instance. (Если клонирование `service.Operation` окажется запутанным, запасной вариант — направлять данные сообщения через scope на каждый instance вместо operation — решается в M2 с доказательствами.)

### 4.4 Вехи (каждая независимо собираема + CI-зелёная)

> **Заметка о порядке.** `Snapshot.Clone()` итерирует `s.Nodes`, вызывая `Clone()` каждого узла, а добавление `Clone()` в интерфейс `flow.Node` вынуждает каждого реализующего существовать сразу — поэтому весь код клонирования приземляется как одна атомарная веха, затем прошивка, затем приёмка.

- **M1 — `Clone()` по всей модели.** Добавить `Clone() flow.Node` в интерфейс `flow.Node`; реализовать на `BaseNode` (shell: id/name, свежие пустые flow, без контейнера), встроенных родителях (`Event`/`catchEvent`/`throwEvent`, `activity`/`task`, `Gateway`) и 5 конкретных типах узлов — конфиг разделён по ссылке, четыре мутируемых при исполнении поля свежие, `ServiceTask` с operation на каждый instance (FR-2/3/5/6); добавить хелпер клонирования ребра пакета `flow` (сохранить id/условие, без вставки в контейнер, FR-4); регенерировать моки. Per-type + хелпер юнит-тесты (клон независим, конфиг разделён, состояние обнулено, контейнер очищен). Сборка зелёная; ещё не прошито в исполнение.
- **M2 — прошить `Snapshot.Clone()` + `instance.New`.** `snapshot.(*Snapshot).Clone()`, зеркалящий node-loop/flow-loop из `snapshot.New` (клонировать узлы через `Clone()`, перевязать flow через хелпер ребра, **переотобразить `defaultFlow` каждого шлюза на клонированное ребро**, разделить неизменяемый заголовок); `instance.New` меняет разделяемый snapshot на свой клон (FR-1); **задокументировать** границу кеинга подписки на события, отложенную в ADR-006 (FR-7); **`-race` тест независимости** — два конкурентных instance одного **простого** процесса (Start→ServiceTask→End), без гонки + независимые `dataPath`/operation/`scope`; все примеры запускаются.
- **M3 — приёмка.** `make ci` зелёный (особенно `-race`); smoke примеров; согласовать **ADR-001 §4.7** + обновить **roadmap**; flip **ADR-009 + SRD-006 → Accepted**.

## 6. Риски и регрессии

- **Запутанность `service.Operation` (FR-5).** Его Items сообщения мутируются при исполнении; если чистый клон на каждый instance труден, откатиться к направлению данных сообщения через scope на каждый instance (решается в M2 с доказательствами). Тест V3 охраняет это.
- **Обратная ссылка `container` (FR-6).** Клонированный узел, всё ещё указывающий на исходный контейнер Process, или перепрошивка, триггерящая `Container().Add`, может провалить валидацию (`sequenceflow.go:122-128/176`). Перепрошивка должна пропускать вставку в контейнер, и клоны не должны сохранять чужой контейнер.
- **Неэкспортируемый `flows` (FR-4).** Перепрошивка должна жить в пакете `flow` (или использовать `AddFlow`); клон вне пакета не может перестроить карту напрямую.
- **Регистрация id event-def (FR-7).** Подтверждено: EventHub кеит ожидающих по `eDef.ID()` в одиночку, поэтому клонированные узлы событий (разделяющие id определений) разделили бы ожидающего. Это изоляция **подписки на события** на каждый instance — семантика доставки событий, принадлежащая **ADR-006**, а не владение состоянием узла этого SRD — и она **отложена** туда. Гонка состояния узла, которую этот SRD убирает, живёт на простом flow (без ожидающих); V4 доказывает это на простом процессе.
- **Налог полноты клонирования.** Новый тип узла должен реализовать `Clone()` корректно или молча разделить состояние — задокументировано как действующее правило (ADR-009 §3); lint/тест мог бы охранять это позже.

## 7. Итог реализации

Приземлено на ветке `feat/node-runtime-state` (от `master`).

**Изменения**

- **M1 — `Clone()` по всей модели.** `flow.Node` получил `Clone() Node`; `flow.BaseNode.CloneShell` (свежая идентичность, пустые flow, без контейнера) + panic-stub `BaseNode.Clone`; хелпер ребра `flow.CloneFlow` (сохранить id/условие, без вставки в контейнер). `Clone()` на 5 исполняемых типах узлов (`StartEvent`, `EndEvent`, `ServiceTask`, `UserTask`, `ExclusiveGateway`) + встроенных родителях — конфиг разделён по ссылке, мутируемые при исполнении поля свежие (`Event.dataPath`, `activity.dataPath`, `UserTask.resChan`, `ExclusiveGateway.scope`), `ServiceTask` с `service.Operation` на каждый instance (FR-2/3/5). Цепочка клонирования неошибочна (форма `Must`, зеркаля `data.Value.Clone`): `Message.Clone`/`Operation.Clone` отбрасывают свои возвраты ошибок.
- **M2 — прошивка.** `snapshot.(*Snapshot).Clone()` зеркалит node-loop/flow-loop из `snapshot.New` (клонировать узлы; перевязать рёбра через `flow.MustCloneFlow`, сохраняя id/условие; переотобразить default flow каждого шлюза на его клонированное ребро через `flow.DefaultFlowHolder` / `Gateway.MustUpdateDefaultFlow`; разделить неизменяемый заголовок). `instance.New` меняет разделяемый snapshot на `s.Clone()` (FR-1); `createTracks`/форки прозрачно используют клон (FR-6 — клоны не несут контейнера).
- **FR-7** задокументирован + отложен в ADR-006 (EventHub кеит ожидающих по `eDef.ID()` в одиночку — семантика доставки событий, не владение состоянием узла).
