## Overview

GoBPM implements a sophisticated event-driven architecture that enables loose coupling between process components. Events flow through the system via the EventHub, which manages event distribution, waiter lifecycle, and processor registration.

The system supports two main event paradigms:
- **Event Propagation**: External events pushed into the system
- **Event Generation**: Internal events generated by waiters based on conditions

## Core Architecture

### Component Hierarchy

```
Thresher (Main Engine)
    ├── EventHub (Event Distribution)
    │   ├── Event Waiters (Condition Monitoring)
    │   │   ├── Timer Waiters (Time-based Events)
    │   │   ├── Signal Waiters (Process Signals)  
    │   │   └── Message Waiters (External Messages)
    │   └── Event Queue (Async Processing)
    └── Event Processors (Event Handlers)
```

### Interface Relationships

```go
// Core interfaces form a cohesive event processing system
type EventProcessor interface {
    foundation.Identifyer
    ProcessEvent(context.Context, flow.EventDefinition) error
}

type EventProducer interface {
    RegisterEvent(EventProcessor, flow.EventDefinition) error
    UnregisterEvent(ep EventProcessor, eDefId string) error
    PropagateEvent(context.Context, flow.EventDefinition) error
}

type EventWaiter interface {
    foundation.Identifyer
    EventDefinition() flow.EventDefinition
    EventProcessors() []EventProcessor
    Process(flow.EventDefinition) error  // For propagated events
    Service(context.Context) error       // For generated events
    Stop() error
    State() EventWaiterState
}
```

## Event Types and Definitions

### Timer Events

Timer events are **generated** by timer waiters based on temporal conditions. They cannot be propagated externally.

```go
// Create timer event definition
timerEventDef := events.MustTimerEventDefinition(
    goexpr.Must(
        nil,
        data.MustItemDefinition(
            values.NewVariable(time.Now().Add(time.Second))),
        func(_ context.Context, ds data.Source) (data.Value, error) {
            return values.NewVariable(time.Now().Add(time.Second)), nil
        }),
    nil, nil)

// Register with processor
err := thresher.RegisterEvent(processor, timerEventDef)

// Timer waiter will automatically generate events when time condition is met
```

**Key Characteristics:**
- Generated internally by timer waiters
- Cannot be propagated via `PropagateEvent()`
- Waiter runs in background via `Service()` method
- Events fired automatically when time conditions are met

### Signal Events

Signal events represent process-level signals and can be both propagated and generated.

```go
// Create signal
signal := &events.Signal{
    BaseElement: *foundation.MustBaseElement(),
    // signal configuration
}

// Create signal event definition
signalEventDef := events.MustSignalEventDefinition(signal)

// Can be propagated externally
err := thresher.PropagateEvent(ctx, signalEventDef)
```

### Message Events

Message events handle external communication and support both propagation and generation patterns.

```go
// Create message event definition
messageEventDef := events.MustMessageEventDefinition(message, nil)

// Support external propagation
err := thresher.PropagateEvent(ctx, messageEventDef)
```

## Event Waiters

Event waiters are components that monitor for specific conditions and notify processors when events should occur.

### Timer Waiter Architecture

Timer waiters follow a unique pattern - they generate events rather than process them:

```go
// Timer waiter's Process method intentionally returns error
func (tw *timeWaiter) Process(eDef flow.EventDefinition) error {
    return fmt.Errorf(
        "timeWaiter doesn't process any EventDefinition (got EventDefinition #%s of type %s)",
        eDef.Id(), eDef.Type())
}

// Events are generated via Service method
func (tw *timeWaiter) Service(ctx context.Context) error {
    // Background monitoring and event generation
    for {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
            if tw.shouldFireEvent() {
                // Generate event for registered processors
                for _, processor := range tw.processors {
                    processor.ProcessEvent(ctx, tw.eDef)
                }
            }
        }
    }
}
```

### Waiter Lifecycle Management

```go
// Create waiter
waiter, err := waiters.NewTimeWaiter(hub, processor, timerEventDef, "waiter-id")
if err != nil {
    return err
}

// Start waiter service in background
go func() {
    err := waiter.Service(ctx)
    if err != nil {
        log.Printf("Waiter failed: %v", err)
    }
}()

// Monitor waiter state
for {
    state := waiter.State()
    if state == eventproc.WSEnded || state == eventproc.WSFailed {
        break
    }
    time.Sleep(100 * time.Millisecond)
}
```

## Event Propagation vs Generation

### Event Propagation Pattern

Used for external events that need to be distributed to registered processors:

```go
// External event received
externalEvent := // ... received from external source

// Propagate to registered processors
err := eventHub.PropagateEvent(ctx, externalEvent)
if err != nil {
    log.Printf("Event propagation failed: %v", err)
}
```

**Flow:**
1. External system generates event
2. Event propagated via `PropagateEvent()`
3. EventHub routes to registered processors
4. Processors handle event asynchronously

### Event Generation Pattern

Used for events that are internally generated based on conditions:

```go
// Waiter monitors conditions
func (w *MyWaiter) Service(ctx context.Context) error {
    for {
        if w.checkCondition() {
            // Generate event internally
            for _, processor := range w.processors {
                err := processor.ProcessEvent(ctx, w.eventDef)
                if err != nil {
                    return err
                }
            }
        }
    }
}
```

**Flow:**
1. Waiter monitors specific conditions
2. When condition met, waiter generates event
3. Event sent directly to registered processors
4. No external propagation needed

## Implementation Patterns

### Event Processor Implementation

```go
type ProcessEventProcessor struct {
    id          string
    processID   string
    eventTypes  map[flow.EventTrigger]bool
}

func (p *ProcessEventProcessor) Id() string {
    return p.id
}

func (p *ProcessEventProcessor) ProcessEvent(
    ctx context.Context, 
    eDef flow.EventDefinition,
) error {
    // Validate event type
    if !p.eventTypes[eDef.Type()] {
        return fmt.Errorf("unsupported event type: %s", eDef.Type())
    }
    
    // Process event based on type
    switch eDef.Type() {
    case flow.TriggerTimer:
        return p.processTimerEvent(ctx, eDef)
    case flow.TriggerMessage:
        return p.processMessageEvent(ctx, eDef)
    case flow.TriggerSignal:
        return p.processSignalEvent(ctx, eDef)
    default:
        return fmt.Errorf("unknown event type: %s", eDef.Type())
    }
}
```

### Event Producer Implementation

```go
type CustomEventProducer struct {
    mu          sync.RWMutex
    processors  map[string]EventProcessor
    eventTypes  map[string]flow.EventDefinition
    eventQueue  chan eventJob
}

func (p *CustomEventProducer) RegisterEvent(
    processor EventProcessor,
    eventDef flow.EventDefinition,
) error {
    p.mu.Lock()
    defer p.mu.Unlock()
    
    key := fmt.Sprintf("%s_%s", processor.Id(), eventDef.Id())
    p.processors[key] = processor
    p.eventTypes[key] = eventDef
    
    return nil
}

func (p *CustomEventProducer) PropagateEvent(
    ctx context.Context,
    eventDef flow.EventDefinition,
) error {
    // Find matching processors
    var targetProcessors []EventProcessor
    
    p.mu.RLock()
    for key, processor := range p.processors {
        if p.eventTypes[key].Type() == eventDef.Type() {
            targetProcessors = append(targetProcessors, processor)
        }
    }
    p.mu.RUnlock()
    
    // Process asynchronously
    for _, processor := range targetProcessors {
        select {
        case p.eventQueue <- eventJob{ctx: ctx, processor: processor, event: eventDef}:
        case <-ctx.Done():
            return ctx.Err()
        default:
            return fmt.Errorf("event queue full")
        }
    }
    
    return nil
}
```

## Testing Event Systems

### Testing Timer Events

Timer events require special handling in tests since they cannot be propagated:

```go
func TestTimerEventProcessing(t *testing.T) {
    // Create thresher and start it
    th, err := thresher.New("test-thresher")
    require.NoError(t, err)
    
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()
    
    err = th.Run(ctx)
    require.NoError(t, err)
    
    // Create mock processor
    mockProcessor := mockeventproc.NewMockEventProcessor(t)
    mockProcessor.EXPECT().Id().Return("test-processor").Maybe()
    
    // Create timer event definition
    timerEventDef := events.MustTimerEventDefinition(
        goexpr.Must(nil,
            data.MustItemDefinition(
                values.NewVariable(time.Now().Add(time.Second))),
            func(_ context.Context, ds data.Source) (data.Value, error) {
                return values.NewVariable(time.Now().Add(time.Second)), nil
            }),
        nil, nil)
    
    // Expect ProcessEvent to be called when timer fires
    mockProcessor.EXPECT().ProcessEvent(
        mock.Anything, 
        mock.MatchedBy(func(ed flow.EventDefinition) bool {
            return ed.Type() == flow.TriggerTimer
        })).Return(nil).Once()
    
    // Register event processor (creates timer waiter)
    err = th.RegisterEvent(mockProcessor, timerEventDef)
    require.NoError(t, err)
    
    // Wait for timer to fire
    time.Sleep(2 * time.Second)
    
    // Verify expectations
    mockProcessor.AssertExpectations(t)
}
```

### Testing Event Propagation

For propagatable events, test both registration and propagation:

```go
func TestEventPropagation(t *testing.T) {
    // Setup similar to timer test...
    
    // Create propagatable event (not timer)
    messageEventDef := events.MustMessageEventDefinition(message, nil)
    
    // Register processor
    err = th.RegisterEvent(mockProcessor, messageEventDef)
    require.NoError(t, err)
    
    // Expect ProcessEvent call
    mockProcessor.EXPECT().ProcessEvent(ctx, messageEventDef).Return(nil).Once()
    
    // Propagate event
    err = th.PropagateEvent(ctx, messageEventDef)
    require.NoError(t, err)
    
    // Allow async processing
    time.Sleep(100 * time.Millisecond)
    
    mockProcessor.AssertExpectations(t)
}
```

## Common Pitfalls

### 1. Trying to Propagate Timer Events

❌ **Incorrect:**
```go
// This will fail - timer events cannot be propagated
timerEvent := events.MustTimerEventDefinition(...)
err := thresher.PropagateEvent(ctx, timerEvent) // ERROR!
```

✅ **Correct:**
```go
// Timer events are generated by waiters
timerEvent := events.MustTimerEventDefinition(...)
err := thresher.RegisterEvent(processor, timerEvent) // Creates waiter
// Waiter will generate events automatically via Service()
```

### 2. Forgetting Async Nature

❌ **Incorrect:**
```go
err := thresher.PropagateEvent(ctx, event)
// Event may not be processed yet!
assert(eventProcessed)
```

✅ **Correct:**
```go
err := thresher.PropagateEvent(ctx, event)
// Allow time for async processing
time.Sleep(100 * time.Millisecond)
// Or use channels/waitgroups for synchronization
```

### 3. Not Handling Waiter Lifecycle

❌ **Incorrect:**
```go
// Waiter created but Service() never called
waiter, _ := waiters.NewTimeWaiter(...)
// Events will never be generated!
```

✅ **Correct:**
```go
waiter, _ := waiters.NewTimeWaiter(...)
go waiter.Service(ctx) // Start waiter service
```

### 4. Ignoring Context Cancellation

❌ **Incorrect:**
```go
func (w *MyWaiter) Service(ctx context.Context) error {
    for {
        // Infinite loop without context checking
        w.checkConditions()
    }
}
```

✅ **Correct:**
```go
func (w *MyWaiter) Service(ctx context.Context) error {
    for {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
            w.checkConditions()
        }
    }
}
```

## Performance Optimization

### Event Queue Sizing

```go
type OptimizedEventHub struct {
    eventQueue   chan eventJob
    workerPool   []worker
    maxQueueSize int
}

func (h *OptimizedEventHub) PropagateEvent(
    ctx context.Context,
    event flow.EventDefinition,
) error {
    select {
    case h.eventQueue <- eventJob{ctx: ctx, event: event}:
        return nil
    case <-ctx.Done():
        return ctx.Err()
    default:
        // Queue full - implement backpressure
        return fmt.Errorf("event queue full, dropping event")
    }
}
```

### Batch Processing

```go
func (h *EventHub) processEvents(ctx context.Context) {
    batch := make([]eventJob, 0, 100)
    ticker := time.NewTicker(10 * time.Millisecond)
    defer ticker.Stop()
    
    for {
        select {
        case job := <-h.eventQueue:
            batch = append(batch, job)
            if len(batch) >= 100 {
                h.processBatch(batch)
                batch = batch[:0]
            }
        case <-ticker.C:
            if len(batch) > 0 {
                h.processBatch(batch)
                batch = batch[:0]
            }
        case <-ctx.Done():
            return
        }
    }
}
```

## Integration Patterns

### With Thresher Engine

```go

type Thresher struct {
    eventHub eventproc.EventProducer
    // ... other fields
}

    processor eventproc.EventProcessor,
    eventDef flow.EventDefinition,
) error {
    return t.eventHub.RegisterEvent(processor, eventDef)
}
```

### With Process Instances

```go
type ProcessInstance struct {
    id             string
    eventProcessor *InstanceEventProcessor
    thresher       *Thresher
}

func (pi *ProcessInstance) Start() error {
    // Register for boundary events
    for _, boundaryEvent := range pi.boundaryEvents {
        err := pi.thresher.RegisterEvent(pi.eventProcessor, boundaryEvent)
        if err != nil {
            return err
        }
    }
    return nil
}
```

## Monitoring and Observability

### Event Metrics

```go
type EventMetrics struct {
    EventsProcessed   prometheus.Counter
    EventsDropped     prometheus.Counter
    ProcessingLatency prometheus.Histogram
    ActiveWaiters     prometheus.Gauge
}

func (m *EventMetrics) RecordEvent(eventType string, duration time.Duration) {
    m.EventsProcessed.WithLabelValues(eventType).Inc()
    m.ProcessingLatency.WithLabelValues(eventType).Observe(duration.Seconds())
}
```

### Health Checks

```go
func (h *EventHub) HealthCheck() error {
    if len(h.eventQueue) > h.maxQueueSize*0.9 {
        return fmt.Errorf("event queue near capacity: %d/%d", 
            len(h.eventQueue), h.maxQueueSize)
    }
    
    failedWaiters := 0
    for _, waiter := range h.waiters {
        if waiter.State() == eventproc.WSFailed {
            failedWaiters++
        }
    }
    
    if failedWaiters > 0 {
        return fmt.Errorf("failed waiters detected: %d", failedWaiters)
    }
    
    return nil
}
```

## Conclusion

The GoBPM event processing system provides a robust foundation for building event-driven business processes. Understanding the distinction between event propagation and generation is crucial for proper implementation and testing.

Key takeaways:
- Timer events are generated, not propagated
- All event processing is asynchronous
- Proper context handling is essential
- Testing requires understanding of event patterns
- Performance considerations matter at scale
